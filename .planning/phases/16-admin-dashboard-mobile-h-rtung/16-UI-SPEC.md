# Phase 16 — UI Design Contract

> Visual and interaction contract for the Admin-Dashboard (`/admin`) and the cross-cutting Mobile-Härtung sweep across Phase 10–15 admin surfaces. Generated by `gsd-ui-researcher` from `16-CONTEXT.md` (D-01..D-22), `REQUIREMENTS.md` (ADMIN-01/02/03 + MOBILE-ADM-01/02/03), `15-UI-SPEC.md` (carry-forward), and direct inspection of `apps/web/components.json` + `apps/web/src/app.css`.

Scope of this contract:

- **A) Admin-Dashboard surface** — single page at `/admin` with a vertical Setup-Completeness checklist of 10 categories plus role-aware login redirect from `/`.
- **B) Mobile-Härtung surface** — global touch-target lift, shared `<DataList>` component with column→card render-prop, and a 375px audit-and-fix sweep across all admin pages from Phases 10–15.

This phase introduces ZERO new design tokens, ZERO new shadcn primitives, and ZERO new color values. Every visual decision below references existing tokens already declared in `apps/web/src/app.css` (carry-forward from Phase 15).

### Touch-target floors (CONTEXT D-17 — global lift)

This is the headline Mobile-Härtung change. All interactive primitives MUST satisfy these floors after Phase 16 ships:

| Primitive | Today (per current source) | Phase 16 floor | Tailwind |
|-----------|---------------------------|----------------|----------|
| `Button` `size="default"` | `h-10` (40px) | `min-h-11` (44px) on `<sm` + keep `h-10` desktop | `h-10 sm:h-10 min-h-11 sm:min-h-10` |
| `Button` `size="sm"` | `h-9` (36px) | `min-h-11` (44px) on `<sm` | `h-9 min-h-11 sm:min-h-9` |
| `Button` `size="lg"` | `h-11` (44px) | unchanged | `h-11` |
| `Button` `size="icon"` | `h-10 w-10` (40×40) | `min-h-11 min-w-11` on `<sm` | `h-10 w-10 min-h-11 min-w-11 sm:min-h-10 sm:min-w-10` |
| `Input` | `h-10` (40px) per `input.tsx:13` | `min-h-11` (44px) on `<sm` | `h-10 min-h-11 sm:min-h-10` |
| Checklist row (Dashboard) | n/a (new) | `min-h-14` (56px) — comfortable on touch + room for icon + 2-line copy | `min-h-14` |
| Mobile-card row (`<DataList>`) | n/a (new) | `min-h-20` (80px) per CONTEXT specifics line 172 | `min-h-20` |

**Note on touch-target floor sub-scale:** `min-h-11` (44px), `min-h-14` (56px), `min-h-20` (80px) are sizing constraints expressing accessibility floors, NOT entries in the spacing scale above. Each value is multiple-of-4 compliant. They are declared here in their own sub-table so that the spacing scale stays clean (4/8/16/24/32/48/64 only). Any class outside this sub-table or the spacing scale above is a drift and a checker fail.

**Rationale for the responsive lift:** desktop pointer accuracy makes 40px targets perfectly usable; tightening desktop to 44px makes UI feel oversized in dense admin tables. The lift is `<sm` only (≤639px). This matches the `useIsMobile()` 640px breakpoint exactly so logic and layout agree.

**Edit strategy** — globally lift `Button` and `Input` defaults inside `apps/web/src/components/ui/button.tsx` + `input.tsx` using the responsive Tailwind classes above. Do NOT introduce a new `mobile`-named variant — keep the existing `size` enum and apply the floor unconditionally. This makes the lift transparent for all 100+ existing call-sites.

### Mobile-card density (CONTEXT specifics line 172)

A `<DataList>` mobile card target is **≤80px tall** with default content. This means: leading icon + 2 lines of body copy + trailing chevron, separated by a 1px border. Padding is `p-4` (16px); content gap is `gap-2` (8px).

### Spacing exceptions

- **Dashboard checklist row** internal layout: `flex items-center gap-4 px-4 py-4` (16px on every axis — fully on-scale `md` token). Visual: leading icon (24×24) → title block (title + status text stacked) → trailing status badge → chevron. The `min-h-14` (56px) touch-target floor is reached naturally: `py-4` (32px total vertical padding) + 24px icon = 56px content height = floor exactly. This is a single horizontal flex row at all viewports — the layout does NOT collapse to two lines on mobile. Status badge MAY shrink to icon-only at `<sm` (see Component Inventory § ChecklistItem).

### `<DataList>` API (CONTEXT D-12 + D-14)

```ts
interface DataListColumn<T> {
  key: string;
  header: string;             // German label, e.g. 'Name', 'Klasse'
  cell: (row: T) => ReactNode;
  className?: string;         // optional cell tailwind class (e.g. text-right)
  /** Hide this column on mobile when card-mode is active. Default: false (renders inside the card). */
  hiddenOnMobile?: boolean;
}

interface DataListProps<T> {
  rows: T[];
  columns: DataListColumn<T>[];
  /** Returns the row's stable identifier for React keys + data-testid suffix. */
  getRowId: (row: T) => string;
  /** data-testid value rendered on every row (table <tr> AND mobile card). */
  getRowTestId?: (row: T) => string;
  /** Required: how each row renders as a single mobile card. Receives the same row data. */
  mobileCard: (row: T) => ReactNode;
  /** Optional: forced viewport mode override. Default: detected via useIsMobile(). */
  mode?: 'desktop' | 'mobile' | 'auto';
  /** Optional empty-state slot. */
  emptyState?: ReactNode;
  /** Optional loading skeleton slot. */
  loading?: boolean;
  loadingSlot?: ReactNode;
  /** Optional row-click handler. When set, each row gets cursor-pointer. */
  onRowClick?: (row: T) => void;
}
```

Desktop rendering: native `<table class="w-full text-sm">` with `<thead>` + `<tbody>`. Header row uses the carry-forward Phase 15 table-header style (`text-xs uppercase tracking-wide text-muted-foreground` — see § Typography "Scoped existing-primitive exceptions"). Body rows: `border-t border-border hover:bg-accent`. Each `<tr>` carries `data-testid` from `getRowTestId(row)`.

Mobile rendering: `<div class="flex flex-col gap-2">` with one `<Card>`-styled wrapper per row from `mobileCard(row)`. Each wrapper is `rounded-lg border bg-card p-4 min-h-20`. The `mobileCard` slot is fully owned by the migrating surface — `<DataList>` does NOT prescribe its layout beyond the wrapper.

`mobileCard` MUST internally include the `data-testid` from `getRowTestId(row)` on its outermost element so E2E selectors work in BOTH modes. The `<DataList>` component re-applies it on the wrapper as a backstop.

**Sort/filter/pagination:** out-of-scope for the `<DataList>` primitive itself per CONTEXT discretion. Each migrating surface keeps its own sort/filter UI ABOVE the `<DataList>`. Pagination strategy (cursor vs offset) is decided per-surface and is a planner detail.
