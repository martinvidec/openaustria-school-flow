## Phase Boundary

**In-Scope (per ROADMAP §Phase 10.5 + scope decisions below):**
- **Räume-Booking-Conflict E2E** (Happy + 409-Conflict) gegen die bestehende `/rooms`-Verfügbarkeits-Grid und `RoomBookingDialog`.
- **Ressourcen-CRUD E2E** (Happy + Error-Path) gegen `/admin/resources`.
- **Imports E2E:** Untis-XML Happy-Path + CSV Happy-Path + ein CSV Error-Path (malformed row), gefahren über `/admin/import` und `ImportWizard`.
- **Solver-Workflow E2E** (trigger → result → publish), inkl. Hinzufügen einer **minimalen Aktivieren-Schaltfläche** auf `/admin/solver`, weil sie für die "publish"-Stufe noch fehlt.
- **Desktop + Mobile-375 Projects** für jede neu/modifizierte Spec, sofern UI-relevant (Imports + Solver sind primär Desktop-Workflows; Booking + Ressourcen müssen mobile-375 abdecken).
- **Phase-Verifikation öffnet das UAT-Tor** wieder gemäß `feedback_e2e_first_no_uat.md` (Erfolgsbedingung aus ROADMAP).

**Out-of-Scope (explizite Descope-Beschlüsse 2026-04-22):**
- **Räume-CRUD UI + E2E** — UI existiert nicht in v1.x; verschoben in eine spätere Phase, die das `/admin/rooms` Page liefert (Kandidat: Phase 14 Solver-Tuning oder eigene admin-rooms-Phase). ROADMAP-Edit + `deferred-items.md`-Eintrag erforderlich.
- **Imports CSV Error-Path-Varianten jenseits "malformed row"** (z. B. invalid Header, oversize file, encoding errors) — bleiben für eine spätere Härtungs-Tranche.
- **Solver-Aktivieren-Bestätigungsdialog** — kein "Aktiven Stundenplan ersetzen?"-Confirm, da explizit gegen Option C entschieden (D-12).
- **Solver-Timefold-Stub-Modus** — Test fährt gegen den **echten** Timefold-Sidecar, kein Mock und keine NestJS-SolverStub-Variante (D-10).
- **Pure-API E2E** als Ersatz für fehlende UIs — wurde sowohl für Räume-CRUD als auch für Solver-Publish abgelehnt (10.4-Präzedenz: "E2E heißt browser-driven").
- **Seed-DB-Enrichment** für per-role Smokes (war 10.4-Deferred-Item Nr. 1) — bleibt deferred.
- **People-CRUD-Restbestände** aus 10.4 (Klassen/Lehrer/Schüler/Fächer) — gehören in Phase 11/12.

ROADMAP-Phase-10.5-Success-Criteria-Zeile "Räume-CRUD + Booking-Conflict" wird zu "Räume-Booking-Conflict (Happy + 409)" abgeändert; "Räume-CRUD" wandert mit Begründung in `.planning/phases/10.5-…/deferred-items.md` und in die Success-Criteria der späteren Räume-Admin-UI-Phase. **Begründung:** Es existiert kein `/admin/rooms` Page in `apps/web` (nur `/rooms` Booking-Grid), analog zum 10.4-People-Blocker B1.
- **D-02:** Solver-"publish" wird durch eine **neue Aktivieren-Schaltfläche** auf `solver.tsx` realisiert (siehe D-11/D-12). Pure-API-Variante abgelehnt (Konsistenz mit 10.4-Linie "E2E ist browser-driven").
- **D-03:** Solver-E2E fährt **echten Timefold-Sidecar** mit kleiner Seed-Datenbasis (D-10). Stubbing/Mocking abgelehnt — ergibt eine echte Integrations-Coverage über die volle Strecke `Web → NestJS → Timefold → Callback → WebSocket → UI`.

### Deliverable 1 — Räume-Booking-Conflict E2E (LOCKED)
- **D-04:** Neue Spec `apps/web/e2e/rooms-booking.spec.ts` (+ optional `rooms-booking.mobile.spec.ts`) deckt:
  - **Happy-Path:** Admin loggt sich ein → navigiert zu `/rooms` → wählt Tag + Periode + Raum im Grid → öffnet `RoomBookingDialog` → trägt `purpose='E2E-BOOK-{Date.now()}'` → submit → erwartet **grünen sonner-Toast** (Text aus `useBookRoom.onSuccess`) und neue Belegung im Grid.
  - **Conflict-Path:** Spec bucht denselben Raum/Tag/Periode ein zweites Mal → erwartet **roten sonner-Toast** mit Conflict-Meldung (HTTP 409 aus `RoomService.bookRoom`) **UND keinen** grünen Erfolgs-Toast (silent-4xx-Guard, Pattern aus 10.2 `silent-4xx.spec.ts`).
- **D-05:** afterEach räumt eigene `E2E-BOOK-*`-Bookings über die API auf (DELETE `/rooms/:id/bookings/:bookingId` falls verfügbar; sonst Prisma-direct-Fixture wie `orphan-year.ts`). Pre-existing Bookings bleiben unangetastet.
- **D-06:** Selektoren strikt **role + accessible-name**, kein `data-testid`. CardTitle-Texte über `getByCardTitle` (10.4-Helper). Toast-Strings werden im Plan aus `useRoomAvailability.ts` extrahiert (Quelltext-Anker, kein Raten).

### Deliverable 2 — Ressourcen-CRUD E2E (LOCKED)
- **D-07:** Neue Spec `apps/web/e2e/admin-resources.spec.ts` (+ Mobile-Variante) deckt:
  - **Happy-Path:** Admin → `/admin/resources` → "Ressource hinzufügen" → Form (`name='E2E-RES-{Date.now()}'`, `resourceType=Beamer`, `quantity=2`) → submit → grüner Toast + Ressource erscheint in Liste.
  - **Edit-Path:** dieselbe Ressource bearbeiten (Quantity ändern) → Toast + neuer Wert sichtbar.
  - **Delete-Path:** löschen → Toast + Ressource verschwindet aus Liste.
  - **Error-Path:** mocked 422 via `page.route()` auf `POST /api/v1/schools/:id/resources` (Pattern aus 10.2 `silent-4xx.spec.ts`) → roter Toast, **kein** grüner Toast, Liste unverändert.
- **D-08:** afterEach DELETE aller `E2E-RES-*` über die API (oder Prisma-direct).

### Deliverable 3 — Imports E2E (LOCKED)
- **D-09a:** Drei Fixture-Dateien werden **eingecheckt** unter `apps/web/e2e/fixtures/imports/`:
  - `untis-happy.xml` — minimaler aber valider Untis-XML-Export (genug für `untis-xml.parser.ts`-Pfad), enthält ≥1 Klasse, ≥1 Lehrer, ≥1 Lesson.
  - `csv-happy.csv` — 5 Schüler-Rows mit Header `firstName,lastName,email,class` (gemäß `TARGET_FIELDS.STUDENTS` in `ImportWizard.tsx`).
  - `csv-malformed.csv` — gleicher Header, eine Zeile mit zu wenig Spalten (Komma-zu-wenig) → triggert Parser-Error im `ImportProcessor`.
- **D-09b:** Spec `apps/web/e2e/admin-import.spec.ts` (Desktop) deckt:
  - **Untis-XML Happy:** Wizard Step 1 (Upload `untis-happy.xml`, fileType=`UNTIS_XML`) → Step 2 wird übersprungen (kein Mapping bei XML) → Step 3 Untis-Preview rendert Counts → Step 4 commit → Step 5 Result-Summary zeigt `success>0`, `errors=0`. Toast grün. Job-Eintrag in `ImportHistoryList` sichtbar.
  - **CSV Happy:** Upload `csv-happy.csv` mit `entityType=STUDENTS` → Step 2 ColumnMapper auto-mappt by header-name → Step 3 Dry-Run-Preview zeigt 5 valid rows → Step 4 commit → Step 5 success summary mit 5 imported.
  - **CSV Error-Path:** Upload `csv-malformed.csv` → Dry-Run-Preview oder Commit liefert mindestens **eine** Validation-Error-Row in Result-Summary; UI rendert die Fehler **explizit** (kein silent-success). Genaue Assertion (Dry-Run vs. Commit) hängt von der bestehenden `ImportResultSummary` ab — Planner inspiziert das Component und entscheidet, was die Spec assertet.
- **D-09c:** Mobile-Project entfällt für Imports (Wizard ist Desktop-only — keine UI-SPEC für Mobile bekannt; falls Wizard auf 375px responsive ist, Mobile-Spec optional, sonst skip mit Kommentar in `deferred-items.md`).
- **D-09d:** afterEach: zugehörige `ImportJob`-Records via API DELETE oder Prisma-direct-Cleanup räumen, damit `ImportHistoryList`-Rauschen kontrolliert bleibt.

### Deliverable 4 — Solver-Workflow E2E + Aktivieren-UI (LOCKED)
- **D-10:** **Real-Solver-Mode**: `POST /timetable/solve` mit `{maxSolveSeconds: 30}` body (Default ist 300). Spec setzt Playwright-`test.setTimeout(60_000)` auf 60s. Tiny-Seed-Datenbasis: das bestehende Seed-Schul-Setup mit ≥1 Klasse, ≥1 Subject, ≥1 Teacher reicht — kein neues Fixture-Schema-Bootstrap.
- **D-11:** **`apps/web/e2e/helpers/global-setup.ts`** wird erweitert um einen **Timefold-Sidecar-Health-Check**: `GET ${process.env.E2E_SOLVER_URL ?? 'http://localhost:8080'}/q/health` mit 5s-Timeout (analog zu API/Vite-Checks aus 10.3-01). Sidecar down → fail-fast mit klarem Label `[E2E-PRECHECK] Timefold sidecar`.
- **D-12:** **Aktivieren-UI auf `apps/web/src/routes/_authenticated/admin/solver.tsx`** (kleine Erweiterung):
  - Nach `solve:complete` (also wenn `lastResult` gesetzt ist): zeige `<Button>Aktivieren</Button>` neben den Result-Werten in der "Letztes Ergebnis" Card.
  - Click → `POST /api/v1/schools/:schoolId/timetable/runs/:runId/activate` → sonner-Toast `'Stundenplan aktiviert'` → **Navigation zu `/timetable`** (per `useNavigate` von TanStack Router) → Button vorher disabled mit "Wird aktiviert…".
  - **KEIN** Bestätigungsdialog (gegen Option "Confirm-dialog" entschieden — User-Wahl 2026-04-22).
  - Fehler-Pfad: roter Toast `'Aktivierung fehlgeschlagen'` mit Error-Detail; **kein** Redirect; Button bleibt verfügbar für Retry.
  - Component-Test optional in `solver.spec.tsx` (Vitest), nicht zwingend (Plan entscheidet).
- **D-13:** Spec `apps/web/e2e/admin-solver.spec.ts` (Desktop) deckt **trigger → result → publish**:
  - Admin → `/admin/solver` → klick "Stundenplan generieren" → Toast `'Stundenplan-Generierung gestartet'` → erwartet via `page.waitForResponse` einen `solve:complete` über die WS-Verbindung (kann auch via Polling auf `lastResult` im DOM beobachtet werden, falls WS-Wait-Pattern brüchig ist; Planner entscheidet, dokumentiert im Plan als `pattern-established`).
  - Wenn `Letztes Ergebnis` gerendert ist: klick "Aktivieren" → grüner Toast → Navigation zu `/timetable` → Spec assertet, dass `getByRole('grid')` auf `/timetable` rendert (Smoke aus 10.3-02).
  - **Error-Path** (sofern realistisch innerhalb 30s reproduzierbar): `POST /timetable/solve` mit invalidem schoolId via APIRequestContext → erwartet 4xx; **oder** alternativ `STOPPED`-State testen indem die Spec während eines laufenden Solves den Stop-Button drückt (falls UI vorhanden). Bevorzugt: Stop-/Abort-Path. Plan dokumentiert die Wahl.
- **D-14:** afterEach: `ConflictException` bei Re-Trigger vermeiden — Spec greift nicht in andere Solve-Runs ein und räumt **keine** anderen Runs weg. Aktivierte Runs bleiben aktiv (kein Rollback nötig — `/timetable` darf den Run-Output zeigen).

```tsx
// in solver.tsx, inside Letztes Ergebnis Card (lastResult exists)
const navigate = useNavigate();
const [isActivating, setIsActivating] = useState(false);

async function handleActivate() {
  if (!schoolId || !lastResult) return;
  setIsActivating(true);
  try {
    const res = await apiFetch(
      `/api/v1/schools/${schoolId}/timetable/runs/${lastResult.runId}/activate`,
      { method: 'POST' },
    );
    if (!res.ok) throw new Error(`Aktivierung fehlgeschlagen (HTTP ${res.status})`);
    toast.success('Stundenplan aktiviert');
    navigate({ to: '/timetable' });
  } catch (error) {
    toast.error('Aktivierung fehlgeschlagen', {
      description: error instanceof Error ? error.message : 'Unbekannter Fehler',
    });
  } finally {
    setIsActivating(false);
  }
}

// Button placement (inside Letztes Ergebnis Card, after the dl):
<Button onClick={handleActivate} disabled={isActivating}>
  {isActivating ? 'Wird aktiviert…' : 'Aktivieren'}
</Button>
```
