## Decisions Made

1. **Prefix cleanup isolation (`E2E-RES-*` desktop vs `E2E-RES-MOBILE-*` mobile).** Rationale: both specs run against the same seed school; if both prefixes collapsed to `E2E-RES-*`, a failed desktop run could leak rows that a mobile `afterEach` would then delete, hiding the desktop failure signal and vice versa. The prefix split is a 10-character cost for a perfect isolation guarantee.
2. **Duplicate constants across specs, not shared module.** Playwright projects run in isolated processes; a `e2e/helpers/resources-contract.ts` module would also need matching cleanup helpers, and coupling the two specs through a shared module would make the desktop-green / mobile-red split harder to reason about. The 10-line duplication is cheaper than the coupling.
3. **Mobile RES-CRUD-03 API-seeds instead of UI-seeds.** The UI create path is already covered by RES-CRUD-01.mobile; spending another ~2s of UI interaction to re-seed before delete would add zero new coverage and 100% runtime cost. `createResourceViaAPI(request, name)` via the admin token is the canonical fast seed path.
4. **Dialog title anchor assertion between open + interact.** Every `click Add/Edit/Delete` is followed by `expect(...heading).toBeVisible()` before the next fill/click. Catches animation-race bugs where a prior dialog's close animation runs past the next open, letting fills land on the wrong form.
5. **Mobile project execution documented as N/A under pre-existing environmental blocker.** Per plan acceptance criteria: *"If not feasible — deferred-items.md updated and this acceptance criterion is marked N/A in SUMMARY."* Parallel-execution rule prohibits me from touching `deferred-items.md` (owned by 10.5-01), so documentation lives here. Chromium-375 emulation proves spec logic is correct.
