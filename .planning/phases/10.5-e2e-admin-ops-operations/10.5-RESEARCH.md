# Phase 10.5: E2E Admin-Ops Operations — Research

**Researched:** 2026-04-22
**Domain:** E2E test authoring for four admin-ops surfaces (Rooms booking, Resources CRUD, Imports wizard, Solver workflow) on an existing Playwright harness — plus a minimal UI gap-fix (Solver "Aktivieren" button).
**Confidence:** HIGH — harness, UI surfaces, and backend endpoints are fully code-verified in this session. One item is MEDIUM: the solver health-check URL in CONTEXT.md has a factual error that needs correction before Plan 04.

## Summary

Phase 10.5 authors **four E2E specs** (plus optional mobile twins) against the existing Playwright harness hardened in 10.4, and ships **one small production-code edit** in `apps/web/src/routes/_authenticated/admin/solver.tsx` to add an "Aktivieren" button that wires the existing `POST /api/v1/schools/:schoolId/timetable/runs/:runId/activate` endpoint (verified at `timetable.controller.ts:121`) to the UI. All four admin surfaces exist in `apps/web/src/routes/_authenticated/` and their backend endpoints exist in `apps/api/src/modules/` — no scope blockers like those in 10.4. The phase is spec-authoring + 30 LoC of UI glue.

**Three findings require planner attention before plans are written:**

1. **CONTEXT.md D-11 health-check URL is wrong.** It specifies `GET http://localhost:8080/q/health` for the Timefold sidecar. The actual solver listens on **port 8081** (`application.properties:6`, `docker-compose.yml:86`, `solver-client.service.ts:16`), and the simpler, already-smoke-tested endpoint is `/health` (`HealthResource.java:12`). Both `/health` and `/q/health` exist (SmallRye auto-enables the Quarkus `/q/` prefix), but Docker's own healthcheck uses `/health`. Plan 04 should use `http://localhost:8081/health` with env override `E2E_SOLVER_URL`.
2. **Toast-string audit must happen at Plan time, not runtime.** Existing hook code reveals string drift between CONTEXT.md and reality — e.g. `useSolverSocket` emits `'Stundenplan-Generierung abgeschlossen'` on `solve:complete`, not a freshly-named toast; `useBookRoom` emits `'Raum erfolgreich gebucht'` (not the shape-anchor `'Raum gebucht'`); `useCreateResource` emits `'Ressource gespeichert'`. Every plan must extract strings **verbatim** from the hook source during plan authoring — trust no string in CONTEXT.md without a code-grep receipt.
3. **Solver Error-Path has a clean contract already.** `DELETE /timetable/runs/:runId/stop` exists (`timetable.controller.ts:110`) and returns 200 with `{runId, status:'STOPPED'}`. Plan 04 can drive the error-path by starting a solve, waiting for `progress` to appear, then hitting `/stop` via APIRequestContext or a UI button (UI button not currently present — see D-13 discretion).

**Primary recommendation:** Write four plans exactly as locked in D-22. Plan 03 (Imports) and Plan 04 (Solver+UI) carry the most risk and should be Wave 2; Plans 01 (Booking) and 02 (Resources) are pure spec authoring and can ship in Wave 1 in parallel. Before Plan 04 starts, the planner MUST correct the health-check URL in D-11 to `http://localhost:8081/health` or verify with the user which of the two Quarkus health endpoints to pin.

### Pattern 2: Error-Path via Network Mocking (inherit 10.2-04)

**What:** `page.route('**/api/v1/...', ...)` fulfills 4xx, spec triggers save, asserts red toast, asserts green toast **never** fires (silent-4xx invariant).

**When to use:** Deliverable 2 Error-Path (mock 422 on `POST /resources`), Deliverable 4 Error-Path alternative (mock 500 on `POST /activate`). NOT for Deliverable 1 Conflict-Path — that uses the **real** 409 path by booking twice (more valuable coverage).

await page.route(

### Pattern 3: Conflict-Path via Real Backend (NEW for 10.5)

**What:** For Deliverable 1, don't mock the 409 — **book the same room/day/period twice** in the same spec. The first POST succeeds (green toast), the second hits the real `ConflictException` at `room.service.ts:122`. This exercises the full Prisma query path that checks `RoomBooking` + active `TimetableRun.TimetableLesson` overlap.

**Why prefer real 409 over mock:**
- Tests the real business-logic invariant ("no double-booking").

### Pattern 6: WS-Complete Detection (CD-02 recommendation)

**What:** To detect `solve:complete` without racing sonner's toast queue, assert on the **"Letztes Ergebnis" CardTitle** rendering — that's the DOM contract for `lastResult` being set, and it's stable.

**Rationale:**
- `useSolverSocket:120-130` sets `setLastResult(event)` and calls `toast.success('Stundenplan-Generierung abgeschlossen', { description: ... })`. Both fire in the same tick.
- `solver.tsx:167` conditional-renders the "Letztes Ergebnis" Card **only** when `lastResult` is truthy.
- 10.2-02 WOCH-01 found that sonner queued toasts can falsely satisfy `getByText` when the next toast arrives late — so CardTitle rendering is a more durable signal than the toast text.

**Recommendation:** Use `await expect(getByCardTitle(page, 'Letztes Ergebnis')).toBeVisible({ timeout: 60_000 })` as the `solve:complete` wait, with a 60s timeout that comfortably exceeds the `maxSolveSeconds: 30` contract plus buffer.

### Anti-Patterns to Avoid

- **Asserting on `'Stundenplan-Generierung abgeschlossen'` toast as the completion signal:** sonner may queue it behind earlier toasts; use CardTitle instead `[VERIFIED: 10.2-02 WOCH-01 deferred-items.md]`.
- **Hard-coding `lastResult.runId` string shapes in spec setup:** the runId is generated server-side; specs should extract it from the DOM (`getByText(/Run ID:/)` adjacent span) or from `page.waitForResponse(POST /solve)` body.
- **Using `getByRole('heading', ...)` for `<CardTitle>` text** — CardTitle is a `<div>`. Use `getByCardTitle` helper (10.4-01).
- **Creating a second `RoomBooking` fixture for the conflict-path via the UI twice in the same test** — would take 2x the UI overhead. Seed the first via API (Pattern 3).
- **Skipping `test.setTimeout(60_000)` on the solver spec** — default 60s is sufficient for maxSolveSeconds=30 + WS roundtrip + activate, but if additional asserts pile up, bump to 90s (D-10 hints 60s).
- **Mocking the Timefold sidecar** — D-03/D-10 explicitly reject stubbing. Real integration coverage is the point.
- **Hitting Vite dev HMR during a spec** — long test runs may see file-watcher reloads. CI runs against `vite preview` on `:4173` via `PLAYWRIGHT_BASE_URL`; local runs use `:5173`. `playwright.config.ts:29` already honors `PLAYWRIGHT_BASE_URL`.
- **Relying on the `'q/health'` path in global-setup without checking:** the actual solver's `/health` is served on `:8081` (a plain JAX-RS resource). CONTEXT.md D-11 says `localhost:8080/q/health` — that's **wrong on both port AND path choice**. Planner MUST fix in Plan 04. See Open Questions #1.
- **Deleting activated runs in afterEach** — D-14 explicitly says activated runs stay active.

## Common Pitfalls

### Pitfall 1: Solver health-check URL is wrong in CONTEXT.md D-11

**What goes wrong:** D-11 specifies `GET http://localhost:8080/q/health` as the fail-fast health probe. Plan 04 implements this verbatim. Global-setup raises `ECONNREFUSED localhost:8080` on every run — because nothing listens on 8080 (Keycloak is on 8080 actually! — the solver is on 8081).

**Why it happens:** CONTEXT.md mixed up the Keycloak port (8080) and the solver port (8081). The `/q/health` path is a Quarkus SmallRye convention and IS available on the solver (via `quarkus-smallrye-health` dep at `pom.xml`), but the more idiomatic fingerprint is the custom `/health` resource that Docker's healthcheck already uses (`docker-compose.yml:90`).

**How to avoid:** Before Plan 04 ships, either (a) update CONTEXT.md D-11 to `http://localhost:8081/health` (docker healthcheck parity) or (b) go with `http://localhost:8081/q/health` (Quarkus convention). The planner picks ONE, cites `[VERIFIED: HealthResource.java:12]` or `[VERIFIED: pom.xml quarkus-smallrye-health dep]`, and sets `E2E_SOLVER_URL=http://localhost:8081` with the path appended in the health-check code.

**Warning signs:** Global-setup log reads `[E2E-PRECHECK] Timefold sidecar health-check failed at http://localhost:8080/q/health: ECONNREFUSED`.

### Pitfall 3: RoomBookingDialog purpose-field label unknown

**What goes wrong:** D-04 says the spec fills `purpose='E2E-BOOK-{Date.now()}'`. The `RoomBookingDialog` component uses `<input>` for purpose but the label text for the field isn't in the first 80 lines read in this research. **How to avoid:** Plan 01 MUST read `apps/web/src/components/rooms/RoomBookingDialog.tsx` full file before writing the spec, and quote the exact `<label>` or `htmlFor` text. Use `getByLabel(<exact>)`.

**Warning signs:** Spec fails with `getByLabel('Verwendungszweck') not found` — try `Zweck` or `Beschreibung`.

### Pitfall 4: ImportWizard Step 2 skipped for Untis XML but NOT for CSV

**What goes wrong:** D-09b says "Step 2 wird übersprungen (kein Mapping bei XML)" for Untis. The ImportWizard code (lines 100-120 read) shows `handleFileSelect` → `uploadMutation` → `setStep(2)`. Step 2 is rendered regardless of type; the behavior branches inside Step 2 (`ImportColumnMapper` for CSV vs `ImportUntisPreview` for Untis). The spec must handle the Step 2 → Step 3 transition for BOTH fileTypes, not skip Step 2 for XML.

**Why it happens:** "Skipped" in D-09b likely means "no column-mapping UI is visible — Step 2 just shows the Untis preview and a Next button". The spec still needs to click "Weiter" or equivalent.

**How to avoid:** Plan 03 reads the full `ImportWizard.tsx` (not just the 120 lines read here) to pin the Step 2 → Step 3 trigger button text for BOTH Untis and CSV. Expect different button labels per branch.

### Pitfall 5: Parallel-worker races on shared `seed-school-bgbrg-musterstadt`

**What goes wrong:** Specs 10.5-01 (booking) and 10.5-03 (import) both write to the same seed school concurrently in worker-parallel mode. Race conditions: import job IDs collide (unlikely — UUIDs), RoomBooking for period 1 monday collides (likely — multiple specs try to book the same deterministic slot).

**Why it happens:** Default Playwright worker count is `os.cpus()/2`. No per-spec DB sandbox exists (10.2 deferred-items.md #3 + 10.4-REGRESSION-REPORT §5.1 both flag this).

**How to avoid:** Each spec uses a **different period number** or a **unique purpose prefix** (`E2E-BOOK-01-`, `E2E-BOOK-02-`). For imports, use a `fileName` suffix per spec to keep ImportJob identification distinct. If races persist, Plan decides `workers: 1` in playwright.config.ts — but this is a phase-level decision, not per-spec, and should be deferred unless empirically needed.

**Warning signs:** Green locally, flaky in CI. 10.4-REGRESSION-REPORT already documented this class of failure under "multi-active backfill".

### Pitfall 6: `RoomAvailabilityGrid` cells lack accessible names

**What goes wrong:** The grid uses `role="grid"` on the container and positional `style={{gridRow, gridColumn}}` for cells — but individual cells don't have `aria-label="Raum X, Periode Y, verfuegbar"` or similar. Specs cannot use `getByRole('gridcell', {name: ...})`.

**Why it happens:** UI-SPEC accessibility was minimal in v1.0 room booking. Visual affordance (green=available) instead of semantic label.

**How to avoid:** Plan 01 has two options: (a) add ARIA labels to RoomRow period cells as a tiny gap-fix (crosses the D-23 "spec-only" constraint — discuss with user), or (b) use a DOM-walk from the "Montag" header → first period column → first available cell via `.filter({hasText:''}).first()` — brittle but respects the "zero production edit" rule for Plans 01-03.

**Warning signs:** Spec cannot deterministically pick a "free slot" cell. Consider asserting via the `/api/v1/schools/:schoolId/rooms/availability` GET response (API-side assertion Pattern 5 from 10.4).

### Pitfall 7: Solver E2E exceeds its budget and stalls the suite

**What goes wrong:** Real-solver-mode with `maxSolveSeconds=30` plus callback latency + activate + redirect can hit 45–50s. If the seed school has many classes/teachers, Timefold takes longer. Test wall time spirals.

**Why it happens:** D-10 sets 30s solve cap, but the whole flow (HTTP 202 → solver → callback → WS → DOM update → Aktivieren → navigate) can add 15-20s. If Timefold's seed problem is large, it returns later.

**How to avoid:** Plan 04 sets `test.setTimeout(90_000)` (60s was D-10's suggestion; add a 30s buffer). If too flaky, post an explicit "solver seed is too big" deferred-item in Plan 04 SUMMARY and consider reducing seed to 2 classes × 3 subjects × 1 teacher × 1 room.

**Warning signs:** Test timeout at 60s; solver completes at 65s but spec already gave up.

### Pitfall 8: Error-path for Aktivieren — invalid runId hits 404 not 500

**What goes wrong:** Plan 04 Error-Path mocks `POST /activate` to 500 via `page.route()`. But D-12 says the button errors show `'Aktivierung fehlgeschlagen'` derived from `error.message = Aktivierung fehlgeschlagen (HTTP ${res.status})`. If mock returns 500, toast reads `Aktivierung fehlgeschlagen (HTTP 500)`. If real invalid runId, activateRun throws `NotFoundException` → HTTP 404 → `(HTTP 404)` in toast.

**Why it happens:** The button's error handler template-literalizes the status code.

**How to avoid:** Plan 04 Error-Path spec asserts on the STEM `'Aktivierung fehlgeschlagen'` (without the status suffix) via `getByText(/Aktivierung fehlgeschlagen/)` — regex match — to avoid coupling to the mock's status code.

## Open Questions

1. **Solver health-check URL — `/health` vs `/q/health` on port 8081**
   - What we know: Both exist. Docker healthcheck uses `/health`. Quarkus SmallRye auto-exposes `/q/health`. CONTEXT.md D-11 says `8080/q/health` — wrong port AND the path choice needs a decision.
   - What's unclear: Is there a user preference for matching docker-compose's probe path, or for using Quarkus idiom?
   - Recommendation: Plan 04 picks `/health` (parity with Docker healthcheck, no management port required). Document decision inline. Escalate to user only if discuss-phase wanted Quarkus idiom.

2. **RoomBooking 409 response body shape**
   - What we know: `RoomService.bookRoom` throws `ConflictException('Dieser Raum ist ... belegt. ...')` with a German message. `useBookRoom` extracts `error.detail` preferentially, falling back to `'Buchung fehlgeschlagen'`.
   - What's unclear: Does the Nest global exception filter serialize `ConflictException.getResponse()` with `detail` or `message`?
   - Recommendation: Plan 01 curl-verifies a deliberate 409 as part of research refinement: `curl -X POST -H "Authorization: Bearer ..." -d '{...}' http://localhost:3000/api/v1/schools/.../rooms/bookings` twice, inspect body.

3. **Untis-XML fixture minimality — will a 20-line hand-written stub pass parser + downstream services?**
   - What we know: `untis-xml.parser.ts` produces `{teachers, classes, rooms, lessons}` arrays; `ImportProcessor.untisXmlToRows` maps these to rows per `entityType`; `validateRow` checks required fields.
   - What's unclear: Does downstream `ImportService.processImportRow` (for non-dry-run commit) require DB-side lookups (e.g. "does teacher shortName MAI already exist?") that would fail if the seed DB doesn't have complementary state?
   - Recommendation: Plan 03 tests the Untis happy-path in **dry-run mode only for validation** — commit-mode may require richer DB state. Escalate to user if real commit-mode on a hand-written fixture is demanded.

4. **Aktivieren-Button error-path realism**
   - What we know: D-12 supports mock-based error assertion. Real 404 (invalid runId) and real 500 (DB down) are both achievable.
   - What's unclear: CONTEXT.md's D-13 says "Error-Path (sofern realistisch innerhalb 30s reproduzierbar)" or Stop-Path — user preference mixed.
   - Recommendation: Plan 04 ships `SOLVER-02` as a mocked-500 error-path (bounded runtime, deterministic). Plan 04 SUMMARY notes that `DELETE /stop` path is deferred to imports-hardening tranche if stop-button UI is also deferred.

5. **Mobile-375 viability for RoomAvailabilityGrid and ResourceList**
   - What we know: Both use `overflow-x-auto` wrappers. `RoomAvailabilityGrid` has `minWidth: 200 + periodCount * 80` — at 10 periods that's 1000px, far wider than 375px viewport. User must horizontally scroll.
   - What's unclear: Is horizontal scroll acceptable UX per UI-SPEC? Or does the grid need a mobile-specific compact view?
   - Recommendation: Plan 01 mobile spec asserts "grid is visible" and "first available cell is clickable after horizontal scroll" — document as `pattern-established` for future room-UI mobile specs. If this fails as a blocker, drop rooms mobile spec to deferred-items.md.
