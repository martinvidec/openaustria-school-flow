## Decisions Made

- **Self-provisioning over seed.ts enrichment:** the seed school ships zero rooms and an empty time-grid. Rather than mutate seed.ts (which would leak test data into UAT and touch Phase 10 territory), the spec provisions its own time-grid + throwaway room via the admin token in `beforeAll`. `afterAll` deletes the room — bookings cascade on the FK. Mirrors the 10.4-01 SCHOOL-03 throwaway-year precedent.
- **Real 409 via `page.route()` rewrite, not mocked 4xx:** Research Pattern 3 (CONTEXT.md D-04) explicitly asked for a genuine backend conflict, not a fabricated 4xx. The chosen approach: API-seed the occupying booking on period 2, open the dialog on still-free period 1, let `page.route()` rewrite the POST body's `periodNumber` from 1 → 2. The backend sees two bookings for the same slot and raises `ConflictException`. The UI's `useBookRoom.onError` gets a real `application/problem+json` body with the verbatim detail string. No double-tab race, no fake 4xx.
- **Different period numbers per test to sidestep Pitfall 5:** ROOM-BOOK-01 uses period 1, ROOM-BOOK-02 seeds period 2. Even if a future `--workers=N` run puts both on the same shard sharing the throwaway room, they never collide.
- **Mobile spec authored rather than deferred-as-not-feasible:** Pitfall 6 / Open Q #5 warned the grid's `minWidth = 200 + periodCount*80` might make clicks unreliable at 375px. Our seeded 2-period grid has `minWidth = 360px`, which fits at 375px without horizontal overflow. `scrollIntoViewIfNeeded()` is belt-and-braces insurance. The spec is structurally identical to the desktop-green logic; the only blocker is the environmental WebKit Bus-error-10, which is class-c (environmental) per the 10.4 three-class failure taxonomy and tracked as deferred-items.md entry #5.

**2. [Rule 1 — Bug fix in approach] Real-409 via `page.route()` rewrite, not mocked fake-4xx**
- **Found during:** Task 2 authoring (first-pass implementation had a confused "double-tab race" narrative)
- **Issue:** The initial spec sketch tried to engineer a race between the grid's re-query and the submit click; more importantly, after the API seed the target cell is already rendered as occupied (Ad-hoc chip), so the dialog cannot be re-opened through normal UI interaction on the same slot.
- **Fix:** Open the dialog on a still-free cell (period 1). `page.route('POST /rooms/bookings', …)` intercepts the outgoing POST and rewrites the body's `periodNumber` from 1 → 2. The real backend sees a duplicate on period 2 and raises `ConflictException`. The UI's `useBookRoom.onError` receives an honest `application/problem+json` response with the verbatim German detail string — sonner renders the red toast.
