### 5.1 FLAGGED — `school-year-multi-active.spec.ts` backfill invariant fails against test-polluted DB

**Failing test:** `school-year-multi-active.spec.ts > backfill invariant — all existing rows after migration are isActive=true`
**Failure:** `expected 1 to be 2`
**Evidence from log:** 2 rows for `seed-school-bgbrg-musterstadt` exist in the live DB — `2025/2026` with `isActive=false` and `2027/2028` with `isActive=true`. The seed file (`apps/api/prisma/seed.ts:358-369`) only creates `2025/2026` as active.
**Root cause:** A prior test run (likely the SCHOOL-03 family, or `screenshots.spec.ts` which previously exercised the activate-year UI) created `2027/2028` as a new active year. Plan 10.4-02's atomic-demote fix then correctly demoted `2025/2026` to `isActive=false` in the SAME transaction. The test's invariant *"every existing row is active"* is valid only against fresh seed data, not against a DB polluted by prior E2E runs on the same shared seed school.

**Classification:** **test-DB-pollution, NOT a Wave 1 code regression.** Plan 10.4-02's atomic-demote is behaving correctly — by design, when a second active year is created, the previous one is demoted. The multi-active spec was authored when only a single-year world existed and has not been updated to reset DB state before it runs its invariant check.

**Recommended follow-up (NOT in scope for 10.4-03):**
- **Option A (preferred):** Add a per-spec DB-reset fixture to `school-year-multi-active.spec.ts` that drops and re-seeds `school_years` rows for the seed school before the invariant check. Most correct; most robust to CI ordering.
- **Option B:** Update the invariant to allow either the single-seed shape OR the atomic-demote aftermath shape (1 active + N demoted). Less brittle to parallel specs but dilutes the semantic check.
- **Option C:** Document as a known flaky-in-isolation test and run it only under `--workers=1` against a fresh DB.
- **Owner:** Testing-infra follow-up. Candidate for Phase 10.5 or a dedicated 10.4-patch mini-plan. Logged to `deferred-items.md` of phase 10.4 (to be authored by the next executor touching the file).

### 5.2 FLAGGED — `e2e/screenshots.spec.ts:172` SCREENSHOT SCHOOL-05 fails because `2027/2028` is now the protected active year

**Failing test:** `screenshots.spec.ts:172:3 SCREENSHOT SCHOOL-05: Orphan-guard error visible on delete attempt`
**Failure:** Timed out at 60s waiting for `button[aria-label="Schuljahr 2027/2028 loeschen"]` to become enabled — the button is `disabled` at the time of the click.
**Root cause:** Same test-DB-pollution as §5.1. `2027/2028` is the current active year in the seed DB (pollution from prior runs). Active years are protected from deletion (SchoolYearService.remove throws ConflictException on `isActive=true`, surfaced in the UI as a disabled button). The screenshot spec picks `.first()` matching the German delete label, which happens to be `2027/2028` — the protected year.
**Evidence:** The spec expects the delete attempt to surface an orphan-guard error banner (e.g. "Schuljahr mit zugeordneten Daten kann nicht gelöscht werden"). Instead the button never becomes clickable because a *different* guard (active-year protection) already gates it.

**Classification:** **test-DB-pollution + precondition drift, NOT a Wave 1 code regression.** Wave 1 did not change `screenshots.spec.ts`, did not change the delete button logic, and did not change the active-year protection. This spec was authored against a fresh seed (1 active `2025/2026`, no `2027/2028`) and breaks against any DB where another year has been created + activated.

**Recommended follow-up (NOT in scope for 10.4-03):**
- **Option A (preferred, mirrors 10.4-02 pattern):** Make the screenshot spec self-sufficient — create its own throwaway `E2E-ORPHAN-{Date.now()}` year with `isActive: false` and attach an orphan dependency to it before the delete attempt. `test.afterEach` cleans up the throwaway. Aligns with the Pattern 2 established in Plan 10.4-02.
- **Option B:** Add a beforeAll/afterAll hook that resets the seed school's `school_years` table to a known shape (1 active `2025/2026`, zero extras) before the screenshot test runs. Broader reach, higher risk of colliding with parallel workers.
- **Option C:** Defer — screenshot specs are only reached when SCHOOL-0X tests pass first; in a clean-DB CI run this wouldn't trip. Document as a known-flaky spec under parallel-worker test pollution.
- **Owner:** Testing-infra follow-up. Closest fit to Phase 10.5 spec-hygiene tranche, or a dedicated 10.4-patch mini-plan paired with §5.1 fix.
