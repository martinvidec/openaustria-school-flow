# Phase 10.4: E2E Admin-Ops People - Research

**Researched:** 2026-04-21
**Domain:** E2E test hardening for admin-ops People surfaces (Klassen / Lehrer / Schüler / Fächer)
**Confidence:** HIGH (harness, API surface), MEDIUM (scope-feasibility — see Section "Blocker Analysis")

## Summary

Phase 10.4 ROADMAP wants E2E coverage for 4 admin CRUD surfaces (Klassen, Lehrpersonen, Schüler, Fächer) including "Stundentafel-Zuordnung", "Werteinheiten-Edit", "Elternlink", and "orphan-safe delete" — each with Happy + Error Path, on desktop + mobile-375.

**Critical finding:** **None of these admin UIs exist in `apps/web` yet.** The admin routes under `apps/web/src/routes/_authenticated/admin/` are `school.settings`, `resources`, `solver`, `substitutions`, `timetable-edit`, `timetable-history`, `import` only. No classes/teachers/students/subjects admin pages exist. Per the ROADMAP, those UIs are scheduled for Phase 11 (Lehrer + Fächer) and Phase 12 (Schüler + Klassen) — and both phases explicitly depend on Phase 10.5 having landed, i.e. they come AFTER 10.4. Phase 10.4 as written cannot test UIs that do not exist.

The v1.0 API surface IS available (controllers exist, DTOs exist, service-layer tests exist) — but the success criteria in ROADMAP ("Klassen-CRUD", "Lehrpersonen-CRUD", etc.) describe **UI** CRUD flows, not pure API E2E. Playwright is configured for browser-driven tests against Vite:5173; pure API tests would be unusual in this codebase.

Furthermore, several success-criteria items map to features that do not exist at all in v1.0 (no ParentController/ParentStudent HTTP endpoints for the "Elternlink", no orphan guards on class/teacher/student/subject DELETE — the services cascade without checking dependencies).

**Primary recommendation:** Planner MUST escalate a scope-realignment decision to the user BEFORE writing plans. Two viable paths exist — pick ONE explicitly. See `## Open Questions` below. Do NOT quietly write plans that implement a different phase (e.g. "10.4 ships Klassen admin UI + tests") without user sign-off — that would be Phase 11/12 done in the wrong order and would violate the ROADMAP dependency graph.

### Pattern 1: Happy-Path CRUD Spec (from 10.2 schuljahre.spec.ts)

**What:** Desktop spec creates a throwaway E2E-prefixed row via the UI, asserts toast, asserts row appears in list, afterEach cleans up via direct API/Prisma.
**When to use:** Any CRUD happy-path on admin surfaces.
**Example:**
```typescript
// Source: apps/web/e2e/schuljahre.spec.ts (10.2-03)
test('YEAR-02: delete a dependent-free year, assert success toast + year disappears', async ({ page }) => {
  await page.goto('/admin/school/settings?tab=years');
  await expect(page.getByRole('heading', { name: 'Schuljahre' })).toBeVisible();

  const name = `E2E-DELETE-YEAR-${Date.now()}`;
  await page.getByRole('button', { name: /Neues Schuljahr anlegen/ }).click();
  await page.getByLabel('Name').fill(name);
  // ... fill form ...
  await page.getByRole('button', { name: 'Anlegen' }).click();
  await expect(page.getByText('Schuljahr angelegt.')).toBeVisible();

  await page.getByRole('button', { name: `Schuljahr ${name} loeschen` }).click();
  await page.getByRole('button', { name: 'Loeschen' }).last().click();

  await expect(page.getByText('Schuljahr geloescht.')).toBeVisible();
  await expect(page.getByText(name, { exact: true })).not.toBeVisible({ timeout: 5_000 });
});
```

### Pattern 2: Error-Path via Network Mocking (from 10.2 silent-4xx.spec.ts)

**What:** Intercept the mutation endpoint via `page.route`, return 4xx/5xx, assert red toast visible AND green toast NEVER fires.
**When to use:** Error-path specs — specifically the silent-4xx invariant and validation errors.
**Example:**
```typescript
// Source: apps/web/e2e/silent-4xx.spec.ts (10.2-04)
await page.route(`**/api/v1/teachers/${teacherId}`, async (route) => {
  if (route.request().method() === 'PUT') {
    await route.fulfill({
      status: 422,
      contentType: 'application/json',
      body: JSON.stringify({ statusCode: 422, message: 'Validierung fehlgeschlagen' }),
    });
    return;
  }
  await route.continue();
});

// ... trigger Save ...
await expect(page.getByText(/konnten nicht gespeichert/)).toBeVisible();
// Critical invariant: green toast never fires
await expect(page.getByText(/gespeichert$/)).not.toBeVisible();
```

### Pattern 3: Orphan-Guard Spec (from 10.2 SCHOOL-05)

**What:** Prisma-direct fixture creates an entity WITH dependencies, spec tries to delete via UI, asserts German "wird noch von X Eintraegen verwendet" error toast, cleanup removes fixture regardless of spec outcome.
**When to use:** Any entity where DELETE should be blocked by FK references.
**Example:**
```typescript
// Source: apps/web/e2e/admin-school-settings.spec.ts:104-138
const fixture = await seedOrphanYear(schoolId);
try {
  // ... UI delete attempt ...
  await expect(page.getByText(/wird noch von .* Eintraegen verwendet/)).toBeVisible();
} finally {
  await cleanupOrphanYear(fixture);
}
```

**PROBLEM:** This pattern REQUIRES that the backend service throws a 409 with a German-message translation when dependencies exist. For classes/teachers/subjects/students, this check **does not exist** in v1.0 — see Blocker Analysis. The pattern works only AFTER the backend gets orphan guards.

### Pattern 5: Persistence Check via API, not UI Reload

**What:** After a mutation, hit the GET endpoint directly with `request.get` + Bearer token and assert the DB state matches — rather than reloading the page.
**When to use:** When the UI's GET endpoint for the mutated resource is missing or returns stale data (happened in 10.2 WOCH-01 because GET `/time-grid` is not implemented).
**Rationale:** UI-reload persistence checks mask data loss if the GET path is broken. API persistence is the canonical source of truth.

### Anti-Patterns to Avoid

- **`getByRole('heading', {name: ...})` for shadcn `CardTitle`**: `CardTitle` renders as `<div>`, not a heading. Use `getByText(...)` instead. `[VERIFIED via 10.3-02 bug fix]`
- **`page.getByText('Unterrichtstage')` when the word appears in both `<h3>` and `<p>` copy**: strict-mode violation. Use `getByRole('heading', {name: 'X', level: 3})` to disambiguate. `[VERIFIED via 10.3 deferred-items.md #1a]`
- **`expect(toHaveURL(...))` immediately after a click without `waitForResponse` or `waitForLoadState`**: Racy with TanStack Router navigation. `[VERIFIED via 10.2 wochentage.spec.ts waitForResponse pattern]`
- **Creating a row with `isActive: true` from the Create dialog when another active row exists**: Returns 500 on school-years due to missing atomic demote in `SchoolYearService.create` (deferred #2). If a similar pattern shows up in 10.4 for other entities, verify the backend's create-as-<flag> path is atomic.
- **Relying on seed-default data to be EXACTLY one row**: The seed already creates 3 teachers, 2 classes, ~30 students. Specs must filter by `name.startsWith('E2E-')` or by `Date.now()` suffix to avoid collision with pre-existing seed rows.
- **Shared school ID hard-coded**: `seed-school-bgbrg-musterstadt` is the canonical seed school — use it as a constant in a shared helper; don't re-query on every test. `[VERIFIED in silent-4xx.spec.ts:36]`
- **Multi-worker races**: 10.2 WOCH-01 + ZEIT-01 both mutate the same seed school, deterministically green only with `--workers=1`. If 10.4 mutates shared rows, expect the same race. Prefer per-spec throwaway entities. `[VERIFIED via 10.2 deferred #3]`

## Common Pitfalls

### Pitfall 1: Phase scope exceeds available UI surface area
**What goes wrong:** ROADMAP success criteria require E2E for UIs (Klassen/Lehrer/Schüler/Fächer admin) that do not exist in apps/web yet. Specs written against selectors-that-don't-exist would fail immediately.
**Why it happens:** Phase 10.4 was inserted as a Tier-3 E2E-hardening phase assuming the UIs would exist — but the feature phases (11/12) that deliver them come AFTER 10.4 in the dependency graph.
**How to avoid:** User decision required BEFORE plans. Options in `## Open Questions`.
**Warning signs:** Planner attempting to write `await page.goto('/admin/classes')` without first verifying the route exists in `apps/web/src/routes/_authenticated/admin/`.

### Pitfall 2: Orphan-guard contract not implemented server-side
**What goes wrong:** Success criteria say "Klassen-CRUD: create, edit, delete (orphan-safe)". The SchoolClass.remove service just cascades — there's no 409-with-German-message if the class has students or classSubjects. `[VERIFIED: apps/api/src/modules/class/class.service.ts:113-117]`
**Why it happens:** v1.0 focused on happy-path CRUD; dependency-check policy was deferred to v1.1 admin UI phases.
**How to avoid:** Either (a) push orphan-guard work into 10.4 as a backend gap-fix task, (b) defer orphan-guard coverage to Phase 11/12 where it naturally belongs, or (c) the spec asserts cascade behavior not orphan-block (changes the test's meaning). Pick one explicitly.
**Warning signs:** Plan text saying "assert German error toast" without first verifying the 409 path exists in the service.

### Pitfall 3: Elternlink has no HTTP surface
**What goes wrong:** Success criterion "Schüler-CRUD: ..., Elternlink — je Happy + Error". CreateStudentDto has no parentId field. No ParentController exists. ParentStudent junction rows are only created by `prisma/seed.ts`, never by HTTP. `[VERIFIED: grep -r ParentController apps/api/src returns nothing]`
**Why it happens:** v1.0 parent relationships were a seed-only concern; Phase 2 didn't need an admin UI.
**How to avoid:** Either (a) add backend gap-fix for `POST /students/:id/parents` as a task inside 10.4, (b) defer Elternlink E2E to Phase 12 (STUDENT-02 Erziehungsberechtigte), or (c) rescope Elternlink to mean "student created with parentId in a single atomic call" and add that field to CreateStudentDto. Pick one explicitly.
**Warning signs:** Plan trying to use `POST /api/v1/parent-students` without verifying such an endpoint exists (it does not).

### Pitfall 4: Werteinheiten-Edit UI surface is ambiguous
**What goes wrong:** Werteinheiten comprises `werteinheitenTarget` (scalar on Teacher) + `employmentPercentage` + `reductions[]` (array of TeachingReduction with type + werteinheiten + description). UI could show any subset. Without a UI, specs can't know which fields to fill.
**Why it happens:** Phase 11 (TEACHER-03) will define the UI. `[VERIFIED: ROADMAP "TEACHER-03: Admin kann Lehrverpflichtung/Werteinheiten editieren"]`
**How to avoid:** If 10.4 does write teacher UI + specs, first draft the UI's field list in a discuss-phase; don't have the planner guess.
**Warning signs:** Specs filling fields like `getByLabel('Werteinheiten')` without citing which component renders that label.

### Pitfall 5: `.mobile.spec.ts` file naming is the only way to enter the mobile-375 project
**What goes wrong:** A spec saved as `admin-classes.spec.ts` with mobile assertions would run under the `desktop` project at 1280x800, not 375x812. Mobile-specific UI (sticky save bar, card layout, drawer nav) would not be exercised.
**Why it happens:** Playwright config uses `testMatch: /.*\.mobile\.spec\.ts$/` for mobile-375 and `testIgnore: /.*\.mobile\.spec\.ts$/` for desktop.
**How to avoid:** EVERY form-editing spec needs BOTH a `.spec.ts` (desktop) AND a `.mobile.spec.ts` (mobile) file per ROADMAP success-criterion. `[VERIFIED: apps/web/playwright.config.ts:37-44]`
**Warning signs:** Plan lists 4 spec files but ROADMAP demands mobile form coverage — count should be 8 files (or form specs should be duplicated with mobile variants).

### Pitfall 6: shadcn/ui `CardTitle` renders as `<div>`, not heading
**What goes wrong:** `getByRole('heading', {name: 'X'})` fails on any `<CardTitle>X</CardTitle>` usage. Hit twice already in this repo (10.2-03 + 10.3-02).
**Why it happens:** shadcn upstream decision — `CardTitle` is a non-semantic `<div className="text-2xl font-semibold">`.
**How to avoid:** Use `getByText('X')` for CardTitle text; reserve `getByRole('heading')` only for real `<h1>`/`<h2>`/`<h3>` elements (e.g. PageShell's h1).
**Warning signs:** Spec fails with "element(s) not found" when the text is clearly visible in the browser. `[VERIFIED: 10.3-02 deviation note]`

### Pitfall 7: Parallel workers race on shared seed rows
**What goes wrong:** Two specs mutating `seed-school-bgbrg-musterstadt.timeGrid` in parallel PUTs cause one to lose — green with `--workers=1`, flaky with default parallelism.
**Why it happens:** The default Playwright worker count is `os.cpus()/2`; no per-spec DB sandbox exists.
**How to avoid:** Option A — use per-spec throwaway entities (create E2E-<timestamp>-class, delete in afterEach). Option B — mark phase-specific projects as `workers: 1`. Option A preferred — more tests fit in the same wall time. `[VERIFIED: 10.2 deferred-items.md #3]`
**Warning signs:** Specs calling PUT on a seeded resource shared across tests.

### Pitfall 8: Seed school has German umlauts in search; normalize case
**What goes wrong:** Searching for "Müller" might miss rows indexed as "Mueller" or vice versa (seed uses "Mueller" without umlauts: `Maria Mueller`). `[VERIFIED: grep Mueller seed.ts]`
**Why it happens:** Historical ASCII-only seed choice.
**How to avoid:** Use ASCII versions in E2E assertion strings; don't introduce umlauts unless you verify the seed has them.

```typescript
export async function seedThrowawayClass(schoolId: string, schoolYearId: string) {

    await prisma.$disconnect();

export async function cleanupThrowawayClass(classId: string) {
  const prisma = new PrismaClient();
  try {
    await prisma.schoolClass.deleteMany({ where: { id: classId } });
  } finally {
    await prisma.$disconnect();
  }
}
```

### afterEach cleanup of E2E-prefixed throwaways

```typescript
// Source: apps/web/e2e/schuljahre.spec.ts:43-69
async function cleanupThrowaway<T extends {id: string; name: string}>(
  request: APIRequestContext,
  listUrl: string,
  deleteUrl: (id: string) => string,
): Promise<void> {
  const token = await getAdminToken(request);
  const listRes = await request.get(listUrl, { headers: { Authorization: `Bearer ${token}` } });
  if (!listRes.ok()) return;
  const rows = (await listRes.json()) as T[];
  const throwaways = rows.filter((r) => r.name.startsWith('E2E-'));
  for (const r of throwaways) {
    await request.delete(deleteUrl(r.id), { headers: { Authorization: `Bearer ${token}` } });
  }
}
```

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|

### Blocker B1: Admin UIs for Klassen / Lehrer / Schüler / Fächer do not exist

**Evidence:**
```
ls apps/web/src/routes/_authenticated/admin/
  import.tsx  resources.tsx  school.settings.tsx  solver.tsx
  substitutions.tsx  timetable-edit.tsx  timetable-history.tsx
```

No `classes.tsx`, `teachers.tsx`, `students.tsx`, `subjects.tsx`, or nested folders for these resources. Component dir `apps/web/src/components/admin/` contains only `school-settings/` and `shared/`. Hooks `useClasses.ts`, `useStudents.ts`, `useSubjects.ts` do not exist (only `useTeachers.ts` — and it is a minimal read-only wrapper for the substitution UI).

**Impact:** Every ROADMAP success criterion in Phase 10.4 (Klassen-CRUD, Lehrpersonen-CRUD, Schüler-CRUD, Fächer-CRUD) requires a UI to test.

**ROADMAP dependency conflict:** Phase 11 says "Depends on: Phase 10 (Schule + Schuljahr müssen vor Lehrer-Anlage existieren)" and delivers TEACHER + SUBJECT UI. Phase 12 says "Depends on: Phase 11 (Fächer + Stundentafel-Vorlagen werden für CLASS-03 benötigt)" and delivers STUDENT + CLASS UI. Both phases logically come AFTER 10.4 in the graph — so 10.4 is ordered before the code it is supposed to test.

### Blocker B2: Orphan guards not implemented server-side

**Evidence:**
- `ClassService.remove`: `await this.prisma.schoolClass.delete({ where: { id } });` (no dependency check) `[VERIFIED apps/api/src/modules/class/class.service.ts:113-117]`
- `TeacherService.remove`: deletes Teacher + Person without checking for TeacherSubject, TeachingReduction, AvailabilityRule, Substitution, etc. `[VERIFIED apps/api/src/modules/teacher/teacher.service.ts]`
- `StudentService.remove`: deletes Person (cascades Student) without checking for GroupMembership, ParentStudent, AbsenceExcuse, ClassBookEntry references.
- `SubjectService.remove`: `this.prisma.subject.delete({ where: { id } });` (no check for ClassSubject, TeacherSubject).

**Impact:** Spec `ClassService-DELETE-ORPHAN-ERROR` (asserts 409 + German toast) cannot pass because the server returns 204. Either (a) add orphan guards as a 10.4 gap-fix, (b) redefine "orphan-safe" as "cascade is OK" (changes test semantics), or (c) defer.

### Blocker B3: Elternlink has no HTTP surface

**Evidence:**
- `grep -r ParentController apps/api/src` → no matches

**Impact:** Spec `STUDENT-PARENT-LINK-HAPPY` cannot hit any real endpoint. Direct Prisma write would work but breaks the "E2E through real API" contract.
