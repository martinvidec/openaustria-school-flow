## Implementation Decisions

### Scope Realignment (LOCKED — User Decision 2026-04-22)
- **Option D (Descope to E2E-Infra) gewählt** nach `10.4-RESEARCH.md`-Analyse der drei Blocker B1/B2/B3
- **Blocker B1 (keine UIs):** People-CRUD kann nicht getestet werden ohne UI → verschoben in 11/12
- **Blocker B2 (keine Orphan-Guards):** Backend-Gap-Fix verschoben in 11/12 (wo Tests für orphan-safe delete geschrieben werden)
- **Blocker B3 (kein Elternlink-Endpoint):** Backend-Gap-Fix verschoben in Phase 12 (natural home: STUDENT-02)

### Deliverable 1 — SCHOOL-02 Spec Fix (LOCKED)
- **Problem:** `page.getByText('Unterrichtstage')` matcht 2 Elemente (Label-Heading + Text im `<p class="text-muted-foreground">`). Strict-mode-Violation seit Plan 10.2-02 das Label zu `<h3>` promovete und einen erklärenden Paragraph mit dem gleichen Wort hinzufügte.
- **Fix:** `page.getByText('Unterrichtstage')` → `page.getByRole('heading', { name: 'Unterrichtstage' })`
- **Scope:** `apps/web/e2e/admin-school-settings.spec.ts`, nur die SCHOOL-02-Assertion
- **Akzeptanz:** SCHOOL-02 grün auf `pnpm --filter web e2e --project=desktop` + `--project=mobile-375`

### Deliverable 2 — SCHOOL-03 Backend Fix (LOCKED)
- **Problem:** `POST /school-years` mit `isActive: true` liefert HTTP 500, wenn bereits ein aktiver Year existiert — weil kein atomic-demote stattfindet. Aus 10.2-03-Decisions als bekannt gedeferrt.
- **Fix:** `SchoolYearService.create` — wenn Payload `isActive: true` und bereits ein active Year existiert, beides innerhalb **einer** Prisma-Transaktion behandeln: aktuellen Year auf `isActive: false` setzen, neuen Year mit `isActive: true` anlegen.
- **Hard-Rule:** Schema-unchanged hier; falls doch schema-touch nötig, MUSS Migration-File geliefert werden (`apps/api/prisma/migrations/...`), kein `prisma db push` (CLAUDE.md-Konvention).
- **Scope:** `apps/api/src/modules/school-year/school-year.service.ts` + Unit-Test `school-year.service.spec.ts` + Regression in `apps/web/e2e/admin-school-settings.spec.ts` SCHOOL-03.
- **Akzeptanz:** SCHOOL-03 zeigt Erfolgstoast „Schuljahr angelegt"; zwei aktive Years können nicht gleichzeitig existieren (DB-Invariante per Test bewiesen); Unit-Test beweist Transaktionalität (Rollback bei Fehler).

### Deliverable 3 — SCHOOL-05 Fixture Fix (LOCKED)
- **Problem:** `apps/web/e2e/fixtures/orphan-year.ts:40` konstruiert `new PrismaClient()` und erwartet `DATABASE_URL` in `process.env` — Playwright-Worker-Subprocesses auf macOS verlieren das Env manchmal.
- **Fix:** Entweder expliziter Datasource-Override (`new PrismaClient({ datasources: { db: { url: process.env.DATABASE_URL ?? '<fallback-loaded-via-dotenv>' } } })`) oder `dotenv.config()` am Fixture-Start vor Prisma-Konstruktion — die robustere der beiden Optionen wird vom Planner gewählt, mit kurzer Begründung im Plan.
- **Scope:** `apps/web/e2e/fixtures/orphan-year.ts` (und ggf. andere Fixtures mit gleichem Pattern — Plan muss grep-Audit enthalten).
- **Akzeptanz:** SCHOOL-05 grün in CI + lokal ohne `source .env` im Shell des Playwright-Runners.

### Deliverable 4 — CardTitle-Harness-Hardening (DECISION DEFERRED TO PLANNER)
- **Problem:** shadcn `CardTitle` rendert als `<div>`. Mindestens 2 Specs sind bereits in die Falle getappt (10.2-03 era + 10.3-02 Task 1). Ohne Fix scheitert jede neue Spec, die auf Empty-State-CardTitle prüft.
- **Option 4a:** Helper `getByCardTitle(pageOrScope, name): Locator` in `apps/web/e2e/helpers/` — non-invasiv, ändert Produktionscode nicht.
- **Option 4b:** `CardTitle`-Komponente zu `<h3>` promovieren (Tailwind/shadcn-Audit nötig, um akzidentielle Layout-Regressionen auszuschließen).
- **Lock:** Planner entscheidet pro/contra im Plan, dokumentiert Entscheidung als kurzer ADR-Absatz, und liefert EINE der beiden Optionen. Eine Mischform ist nicht erlaubt.
- **Akzeptanz:** Mindestens eine bestehende Spec (z. B. roles-smoke.spec.ts oder eine der SCHOOL-Specs) verwendet das neue Helper oder die neue `<h3>`-Struktur und bleibt grün.

### Deliverable 5 — Gesamt-Regression (LOCKED)
- Alle existierenden desktop E2E-Specs, inkl. der per-role Smokes aus 10.3-02, müssen nach den 4 Änderungen grün bleiben.
- Die 3 bisher roten Specs (SCHOOL-02, SCHOOL-03, SCHOOL-05) müssen jetzt grün sein.
- Mobile-375-Projects müssen für SCHOOL-02-Assertion ebenfalls laufen.

### Deliverable 6 — ROADMAP-Edit (bereits erledigt, Plan muss verifizieren)

```typescript
async create(dto: CreateSchoolYearDto): Promise<SchoolYear> {
  return this.prisma.$transaction(async (tx) => {
    if (dto.isActive === true) {
      await tx.schoolYear.updateMany({
        where: { schoolId: dto.schoolId, isActive: true },
        data: { isActive: false },
      });
    }
    return tx.schoolYear.create({ data: dto });
  });
}
```
