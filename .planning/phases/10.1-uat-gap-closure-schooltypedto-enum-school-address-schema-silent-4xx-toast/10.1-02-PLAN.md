Shared SCHOOL_TYPES (packages/shared/src/schemas/school.schema.ts lines 7-18 — SOURCE OF TRUTH for the HTTP contract):
```typescript
export const SCHOOL_TYPES = ['VS', 'NMS', 'AHS', 'BHS', 'BMS', 'PTS', 'ASO'] as const;
```

NOTE the asymmetry: Prisma intentionally keeps MS/AHS_UNTER/AHS_OBER for legacy row compatibility
(per schema.prisma comments), but the UI + shared DTOs only advertise the 7 active values.
The HTTP DTO should follow SCHOOL_TYPES, not Prisma — we do NOT want new schools to be created
with the 3 legacy values.

Current broken DTO (apps/api/src/modules/school/dto/create-school.dto.ts lines 8-14):
```typescript
enum SchoolTypeDto {
  VS = 'VS',
  MS = 'MS',
  AHS_UNTER = 'AHS_UNTER',
  AHS_OBER = 'AHS_OBER',
  BHS = 'BHS',
}
```
This REJECTS 'NMS', 'AHS', 'BMS', 'PTS', 'ASO' with 422 — the bug.

UpdateSchoolDto (apps/api/src/modules/school/dto/update-school.dto.ts — already correct):
```typescript
export class UpdateSchoolDto extends PartialType(CreateSchoolDto) {}
```
`PartialType` preserves the @IsEnum decorator from the parent, so fixing the parent automatically fixes the child.
</interfaces>

<implementation-approach>
Two viable approaches:

Approach A (preferred — DRY, single source of truth):
  Import `SCHOOL_TYPES` from `@schoolflow/shared` and generate the enum at module scope.
  class-validator's `@IsEnum()` decorator accepts a plain object literal of string values as well as a
  TS enum, so we can pass `SCHOOL_TYPES` directly — but NestJS Swagger's `@ApiProperty({ enum })` also
  accepts a readonly string array, so the approach is safe.

Approach B (fallback — inline the 7 values):
  Rewrite the `enum SchoolTypeDto` block verbatim with 7 Phase-10 values, no import.
  Simpler but duplicates the list across two packages and risks future drift.

USE APPROACH A. Rationale: the UAT-FINDINGS §Bug 1 fix recommendation explicitly says
"oder besser: aus SCHOOL_TYPES in @schoolflow/shared importieren — Single Source of Truth".
Confirm `@schoolflow/shared` is already a dependency of `apps/api/package.json` before editing
(it is used in Phase 10 plans and RESEARCH.md). If for some reason the import fails at build time
in the API context (e.g. CJS/ESM interop issue with the apps/api tsconfig), fall back to Approach B
and record the reason in the SUMMARY.
</implementation-approach>
