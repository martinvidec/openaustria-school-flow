## 1. Prisma 7 Schema Migrations

### 1.1 Add `School.abWeekEnabled`
Schema diff:
```prisma
model School {
  // ...existing fields...
  abWeekEnabled Boolean @default(false) @map("ab_week_enabled")
}
```
Migration is trivial (`ALTER TABLE schools ADD COLUMN ab_week_enabled BOOLEAN NOT NULL DEFAULT false`); generated by `npx prisma migrate dev --name 10_add_school_ab_week_enabled`. No backfill required — default covers existing rows.

Confidence: HIGH [VERIFIED: schema inspection]

### 1.2 SchoolYear multi-active migration (partial unique index)
**Prisma 7 partial unique index support:** Prisma's native `@@unique` does NOT accept a `WHERE` clause. Declarative partial indexes remain unsupported in Prisma 7's schema language (see tracking issue prisma/prisma#3076 still unresolved as of 2026-Q1). Workaround: keep the schema declaration honest but add the partial index via raw SQL in the migration file.

Schema diff (drop `@unique`, add `isActive` + plain index):
```prisma
model SchoolYear {
  id       String @id @default(uuid())
  schoolId String @map("school_id")         // no more @unique
  school   School @relation(fields: [schoolId], references: [id], onDelete: Cascade)

  name          String
  startDate     DateTime @map("start_date")
  semesterBreak DateTime @map("semester_break")
  endDate       DateTime @map("end_date")
  isActive      Boolean  @default(false) @map("is_active")

  holidays       Holiday[]
  autonomousDays AutonomousDay[]

  @@index([schoolId])
  @@map("school_years")
}
```

Migration SQL (hand-edited after `npx prisma migrate dev --create-only --name 10_school_year_multi_active`):
```sql
-- Step 1: drop existing unique constraint/index
ALTER TABLE "school_years" DROP CONSTRAINT IF EXISTS "school_years_school_id_key";
DROP INDEX IF EXISTS "school_years_school_id_key";

-- Step 2: add isActive column (default false)
ALTER TABLE "school_years" ADD COLUMN "is_active" BOOLEAN NOT NULL DEFAULT false;

-- Step 3: backfill — every existing row becomes the active year for its school
UPDATE "school_years" SET "is_active" = true;

-- Step 4: non-unique lookup index
CREATE INDEX "school_years_school_id_idx" ON "school_years" ("school_id");

-- Step 5: partial unique index — at most one active year per school
CREATE UNIQUE INDEX "school_years_active_per_school"
  ON "school_years" ("school_id") WHERE "is_active" = true;
```

**Recommendation: TWO separate migrations** (not one combined):
1. `10_add_school_ab_week_enabled` — pure `ALTER`, auto-generated by `prisma migrate dev`, zero risk.
2. `10_school_year_multi_active` — `--create-only` flag, hand-edited SQL, requires backfill.

Rationale: blast-radius isolation (schema-only change can't be tainted by data-migration failure); roll-forward granularity; prisma drift-detection cleaner when raw-SQL file is self-contained.

**After schema edit, regenerate Prisma client:** `npx prisma generate` (pure-TS generator in Prisma 7 — fast).

Confidence: HIGH [CITED: Prisma 7 migration docs, `--create-only` flag]

### 1.3 Data backfill strategy
Inside migration 2's SQL (above, Step 3). Existing schema has `SchoolYear.schoolId @unique`, so each school has at most one SchoolYear — `UPDATE ... SET is_active = true` is safe: at most one row per school becomes `true`, preserving the new partial-unique invariant. No JS-side backfill script required.

**Seed update required:** `apps/api/prisma/seed.ts:340-342` creates a SchoolYear with `findUnique({ where: { schoolId } })`. After migration, `schoolId` is no longer a unique field — that call breaks. See §7.1.

Confidence: HIGH [VERIFIED: existing @unique constraint guarantees backfill safety; seed.ts inspected]

## 4. Period Editor Mechanics

### 4.1 Drag library decision
`@dnd-kit/core ^6.3.0`, `@dnd-kit/sortable ^10.0.0`, `@dnd-kit/utilities ^3.2.0` are ALREADY installed in `apps/web/package.json`. Decision: **use dnd-kit/sortable**. Compatible with React 19, accessible (keyboard reorder via arrow keys + space), small bundle, supports both row and card layouts with a single `SortableContext`.

Canonical usage:
```tsx
<DndContext onDragEnd={handleDragEnd} collisionDetection={closestCenter}>
  <SortableContext items={periods.map(p => p.id)} strategy={verticalListSortingStrategy}>
    {periods.map(p => <SortablePeriodRow key={p.id} period={p} />)}
  </SortableContext>
</DndContext>
```
`SortablePeriodRow` uses `useSortable({ id })` → drag handle gets `attributes + listeners` (44px target at `md:w-8 h-11` → `md:h-8` desktop / `h-11` mobile).

Confidence: HIGH [VERIFIED: package.json lines 20-22]

### 4.2 Time input approach
No mask library in the project. Grep for `type="time"` in `apps/web/src` returns no active usage yet. Phase 4 timetable-edit works on whole periods (not times), so no precedent exists.

**Recommendation:** native `<input type="time">` wrapped in shadcn `<Input>` (`type="time"` prop passthrough). Modern iOS/Android Safari render a native time picker; desktop Firefox/Chrome show a stepper. Controlled via RHF `register('startTime')` returns `HH:mm` strings → matches the Zod regex `/^\\d{2}:\\d{2}$/` directly. No parsing needed.

Computed duration: derived with `date-fns` (already in package.json) — `differenceInMinutes(parse(endTime,'HH:mm',new Date()), parse(startTime,'HH:mm',new Date()))`. Read-only display.

iOS Safari pitfall: at 375px, the native time picker renders a full-screen wheel — 44px min-height on the Input element (`className="h-11"`) ensures tap target compliance with MOBILE-ADM-02.

Confidence: MEDIUM [recommendation; no direct precedent in codebase]

### 4.3 Dense table → stacked cards at 375px
No existing precedent for "responsive editable table" in apps/web. Phase 10 establishes it.

Pattern: single component `<PeriodsEditor>` renders:
- `md:table` layout for desktop (`<table className="hidden md:table">`)
- card-stack for mobile (`<div className="md:hidden space-y-3">` with one `<Card>` per period, drag-handle + delete in card header, fields full-width below)

Both layouts share the same `SortableContext` — dnd-kit is layout-agnostic. Document this as the v1.1 admin-table mobile-adaptation pattern (reusable for Teachers table Phase 13, Classes table Phase 12, etc.).

## 5. Destructive-Edit Protection

### 5.1 TimeGrid → TimetableRun relationship
Critical finding: `TimetableRun` has **no direct FK to TimeGrid or Period**. It only stores `schoolId` (line 684) and a set of `TimetableLesson` children that reference `periodNumber: Int` (not a FK to `Period.id`).

```prisma
model TimetableLesson {
  runId          String
  periodNumber   Int    @map("period_number")   // just a number, not an FK
  dayOfWeek      DayOfWeek
  ...
}
```

**Consequence:** There is no referential integrity to break when TimeGrid is edited. Partial edits (changing a period's `startTime`/`endTime`/`breakAfter`) are always safe at the DB level. What the warn-dialog in D-13 protects is **semantic integrity**: if existing TimetableRun has `periodNumber=5` and the admin removes period #5 from the grid, the solved lessons become orphan schedule slots.

### 5.2 Server-side guard location
New endpoint: `PUT /api/v1/schools/:schoolId/time-grid` (or nested under existing school controller as `PUT /schools/:id/time-grid`).

Guard logic lives in a new `school-time-grid.service.ts` (keep `school.service.ts` untouched for blast-radius isolation):
```ts
async updateTimeGrid(schoolId: string, dto: UpdateTimeGridDto, opts: { force: boolean }) {
  const activeRuns = await this.prisma.timetableRun.findMany({
    where: { schoolId, isActive: true },
    include: { lessons: { select: { periodNumber: true, dayOfWeek: true } } },
  });
  const impactedRuns = activeRuns.filter(run => this.hasImpact(run.lessons, dto));
  if (impactedRuns.length > 0 && !opts.force) {
    throw new ConflictException(problemDetails({
      type: '/errors/time-grid-conflict',
      title: 'Zeitraster-Änderung betrifft aktive Stundenpläne',
      detail: `${impactedRuns.length} aktiver Stundenplan verwendet dieses Zeitraster.`,
      status: 409,
      'impacted-runs': impactedRuns.map(r => ({ id: r.id, createdAt: r.createdAt })),
    }));
  }
  // proceed...
}
```
`hasImpact` compares old vs new period set: any removed `periodNumber` that appears in `run.lessons.periodNumber` is an impact. Changed `startTime`/`endTime` alone is NOT an impact (period identity preserved).

Client retries with `?force=true` if admin chooses "Nur speichern" or "Speichern + Solver neu starten" in the warn-dialog. The "Solver neu starten" branch additionally enqueues a BullMQ `solver.rerun` job (uses existing queue infrastructure).

### 5.3 UI warn-dialog flow
Dialog content:
> **X bestehende Stundenpläne verwenden dieses Zeitraster.**
> Änderungen können Kollisionen verursachen.

Options (match D-13 verbatim): `[Nur speichern]` (force=true, no rerun), `[Speichern + Solver neu starten]` (force=true, enqueue rerun), `[Abbrechen]` (close dialog, revert UI state).

Triggered by a 409 Conflict response from the initial save — RFC 9457 body's `impacted-runs` count drives the dialog text.

Confidence: HIGH [VERIFIED: schema structure, TimetableLesson.periodNumber is Int]

## 8. Pitfalls

- **iOS Safari `<input type="time">` at 375px** — native picker is full-screen modal; ensure the Input has `className="h-11"` to hit 44px tap target.
- **Prisma 7 partial index syntax** — declarative `@@unique` with `where` is still unsupported; keep the SQL in migration file only. Do NOT search Rust-engine docs.
- **`findUnique({ schoolId })` in `seed.ts`** — breaks immediately after migration 2 runs; seed fix must be in the same commit as the migration.
- **TimetableRun creation sites** — School-level `abWeekEnabled` is only the DEFAULT; code that creates new TimetableRuns must copy the value at creation time (the read side in `teacher-absence.service` is unchanged).
- **TanStack Query cache invalidation** — Pro-Tab-Save means 3 separate queries (`['school', id]`, `['time-grid', schoolId]`, `['school-years', schoolId]`). Each mutation must `queryClient.invalidateQueries({ queryKey: [...] })` with the specific key — avoid blanket `invalidateQueries()`.
- **Zustand `useSchoolContext` empty-flow timing** — the Stammdaten-Tab Inline-Create must call the store's setter AFTER the POST returns a 201 body, BEFORE the Zeitraster/Schuljahr/Optionen tabs become enabled. Otherwise `schoolId` is undefined in the other tabs' queries. Disable non-Stammdaten tabs (`<TabsTrigger disabled={!schoolId}>`) until the store has a schoolId.
- **Radix `Select` inside a Dialog** — known z-index stacking issue; use `Select.Portal` + `Dialog.Portal` correctly. Applies to the Unsaved-Changes Dialog if it contains selects (it shouldn't — just 3 buttons).
- **`formState.isDirty` reset on successful submit** — RHF needs explicit `reset(data)` call with the new server state after a successful save so `isDirty` returns to false; forgetting this leaves the blocker permanently armed.
