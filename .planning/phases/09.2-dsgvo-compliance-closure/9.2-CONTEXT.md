# Phase 9.2: DSGVO Compliance Closure - Context

**Gathered:** 2026-04-09
**Status:** Ready for planning
**Mode:** Gap closure (audit-driven)

<domain>
## Phase Boundary

Extend Phase 2 DSGVO services (export, deletion, retention) to cover personal data added by Phases 5-8. The original Phase 2 implementation only handles entities that existed at that time (Person, Teacher, Student, Parent, ConsentRecord, AuditEntry). Subsequent phases added personal data that needs to be exported on Art. 15 requests, anonymized on Art. 17 requests, and pruned per retention policy.

</domain>

<decisions>
## Implementation Decisions

### DSGVO-04: Data Export (Audit Finding 4)
- **D-01:** Extend `apps/api/src/modules/dsgvo/export/data-export.service.ts` to query and include the following Phase 5-8 tables for the requested person:
  - **Phase 5:** AttendanceRecord (where studentId), GradeEntry (where studentId), AbsenceExcuse (where studentId or createdById), StudentNote (where studentId)
  - **Phase 6:** Notification (where userId = person.keycloakUserId)
  - **Phase 7:** Message (where senderId), MessageRecipient (where userId)
  - **Phase 8:** Homework (where assignedById = person), Exam (where createdById = person), CalendarToken (where userId)
- Add a section to the JSON/PDF export for each new data category. Maintain existing PDF 50-entry limit per category to prevent oversized documents.

### DSGVO-05: Deletion Anonymization (Audit Finding 6)
- **D-02:** Extend `apps/api/src/modules/dsgvo/deletion/data-deletion.service.ts` `anonymizePerson()` transaction to handle the foreign keys from Phase 5-8 tables:
  - **Replace user IDs with anonymous placeholder** for: Notification.userId, Message.senderId, MessageRecipient.userId, AbsenceExcuse.createdById, GradeEntry references where applicable
  - **Set message content to '[Geloeschte Nachricht]'** for Message.body where senderId matches
  - **Delete (cascade)** any rows where the person is the only meaningful owner: PushSubscription, CalendarToken, SisApiKey
- Use deterministic anonymous counter pattern from Phase 2 D-15 (hash of person ID).

### DSGVO-06: Retention Cron (Audit Finding 5)
- **D-03:** Extend `apps/api/src/modules/dsgvo/retention/retention.service.ts` `checkExpiredRecords()` to perform actual deletion for the categories that currently only report counts:
  - **noten** (60yr): Delete GradeEntry rows older than retention period
  - **anwesenheit** (5yr): Delete AttendanceRecord rows older than retention period
  - **kommunikation** (1yr): Delete Message + MessageRecipient rows older than retention period (cascade)
- Use Prisma transaction with deleteMany. Log count of deleted rows per category.

### Claude's Discretion
- Exact placeholder strings for anonymized fields
- Whether to preserve message structure (with anonymized author) vs full delete
- Order of deletion in retention cron to satisfy FK constraints
- Test stub coverage strategy (additive to existing dsgvo specs)

</decisions>
