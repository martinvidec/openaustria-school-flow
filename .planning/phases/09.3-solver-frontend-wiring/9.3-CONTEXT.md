# Phase 9.3: Solver Frontend Wiring - Context

**Gathered:** 2026-04-09
**Status:** Ready for planning
**Mode:** Gap closure (audit-driven)

<domain>
## Phase Boundary

Wire the existing /solver Socket.IO namespace into the frontend with admin solve trigger UI and real-time progress display. Backend (Phase 3) emits solve:progress and solve:complete events; this phase adds the frontend consumer to make TIME-06 fully verifiable.

</domain>

<decisions>
## Implementation Decisions

### Audit Finding 3: /solver Socket.IO Has No Frontend Consumer
- **D-01:** Add `createSolverSocket(jwt)` factory to `apps/web/src/lib/socket.ts` following the same pattern as createTimetableSocket / createNotificationSocket.
- **D-02:** Create `useSolverSocket` hook in `apps/web/src/hooks/useSolverSocket.ts` mounted at the admin solver page level (not _authenticated layout — this is page-scoped). Listens for `solve:progress` (updates state with percent + score) and `solve:complete` (invalidates timetable queries + toast).
- **D-03:** Create admin solve trigger page at `apps/web/src/routes/_authenticated/admin/solver.tsx`. Components: school selector, "Stundenplan generieren" button (admin only), live progress bar, score display, last result summary.
- **D-04:** Wire useTimetableRuns hook (already exists from Phase 3) to display run history alongside live solve trigger.
- **D-05:** Add sidebar entry "Stundenplan-Generator" in admin section, visible only to Admin/Schulleitung roles.

### Claude's Discretion
- Exact progress bar styling (use existing shadcn patterns from Phase 3)
- Whether to show constraint score chart or just numeric values
- Result display layout
- Error handling for failed solve runs

</decisions>
