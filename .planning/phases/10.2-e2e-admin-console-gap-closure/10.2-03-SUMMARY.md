## Decisions Made

- **Orchestrator Option A (ship 2/3, defer YEAR-01):** Right call — an E2E plan cannot build missing UI without violating the test-authoring-only scope boundary. Shipping honest 2/3 coverage with a documented gap is better than skipping YEAR-01 silently.
- **YEAR-03 via Aktivieren button, not create-as-active dialog:** The plan's key_link explicitly names `POST /school-years/:yearId/activate via admin clicks Als aktiv setzen on a non-active year` — this is the admin's real daily flow. The create-as-active dialog is a convenience the admin uses *once per school year creation*; the switch is the recurring operation.
- **Single-active invariant at API layer, not DOM:** `SchoolYearCard` renders both an `Aktiv` badge and an `Aktivieren` button. Even with `{ exact: true }`, DOM-level counting is fragile across icon/text refactors. API-layer assertion (`filter(y => y.isActive)`) is stable.
- **Copy corrections vs plan draft:** The plan's draft had `Als aktiv setzen` and `Schuljahr aktualisiert` as stand-in copy. Actual UI surface uses `Aktivieren` (SchoolYearCard.tsx:68) and toast `Aktives Schuljahr gewechselt.` (useSchoolYears.ts:80). Spec matches reality, not the plan's approximations.
