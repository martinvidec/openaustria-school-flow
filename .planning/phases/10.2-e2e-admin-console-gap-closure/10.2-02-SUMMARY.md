## Decisions Made

1. **FIX over DEFER for the UX gap** — rationale in wochentage-decision.md. The affordance is present (SCHOOL-02 proves it) but visually subordinate; a 9+/9- LoC promotion fits the "smallest possible change" rule and does not require a redesign. A list-with-add-button pattern would be wrong because `SCHOOL_DAYS` is fixed by the Austrian school calendar and not user-creatable.
2. **Persistence via GET /schools/:id, not UI reload** — GET /time-grid is 404 (deferred-items #2); a page reload cannot read the persisted schoolDays back into the UI because `useTimeGrid` swallows the 404 and keeps `data=null`, so the useEffect hydration never runs. This mirrors the ZEIT-01 decision from 10.2-01.
3. **waitForResponse over toast-text assertion for multi-save specs** — Sonner queues `Aenderungen gespeichert` toasts across consecutive saves within a single test. A plain `getByText(/Aenderungen gespeichert/)` would see the prior save's toast and short-circuit BEFORE the current save's PUT fires, producing a false-positive persistence check. The Network-layer wait is fully deterministic and survives toast queueing. Recorded as a new pattern for future Playwright specs exercising multiple consecutive saves.
4. **Drive bootstrap through the UI, not the API** — The plan allows either, but clicking `Periode hinzufuegen` + `Unterrichtstag Mo` + `Speichern` exercises the real PeriodsEditor + Toggle code path and therefore gives the spec coverage of more of the UI surface than a bare API seed would. Cost: one extra PUT per test run.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 — Blocking] Bootstrap step required for test to run against the 404-GET environment**

- **Found during:** Task 2 first test run.
- **Issue:** The plan's Task 2 action draft assumed the form would hydrate with the school's current time-grid on mount. In this codebase, `GET /time-grid` is 404, so `useTimeGrid` returns null and the form mounts with `periods=[]` and `schoolDays=[]`. Both client-side validators (`TimeGridSchema.safeParse`) then short-circuit the Save click with `Mindestens eine Periode erforderlich` or `Mindestens ein Unterrichtstag erforderlich` — the PUT never fires and the test fails at `getByText('Aenderungen gespeichert')`.
- **Fix:** Added a conditional bootstrap block (`if needsPeriod add; if needsMo click; if either → save and wait for PUT 200`) that drives the form to a minimum-valid state before the Saturday-specific assertion. Because PUT `/time-grid` does full replace (`tx.deleteMany` + `createMany`), the bootstrap is idempotent across re-runs.
- **Files modified:** apps/web/e2e/wochentage.spec.ts
- **Verification:** `pnpm exec playwright test --grep WOCH-01 --project=desktop` green on first-ever run AND on re-run (no fixture cleanup needed).
- **Committed in:** 80e70ec (Task 2 commit).

**2. [Rule 1 — Bug] waitForResponse replaces toast-text check for second/third saves**

- **Found during:** Task 2 second test run (after the bootstrap fix).
- **Issue:** With the bootstrap save + Saturday save back-to-back, `getByText(/Aenderungen gespeichert/)` matched the bootstrap toast's residue (sonner default duration 4s, the Sa-save fires much faster than that) and reported "visible" BEFORE the Sa-save PUT actually fired. `readSchoolDays()` then observed the pre-Sa-save state and the persistence assertion failed. Logged the outgoing PUT body during debug — confirmed the PUT body for the Sa-save did contain `schoolDays:["MONDAY","SATURDAY"]` and the response was 200, but the API read had raced ahead of the PUT.
- **Fix:** Replaced every `await expect(page.getByText(/Aenderungen gespeichert/)).toBeVisible()` in WOCH-01 with `const r = page.waitForResponse(PUT /time-grid match); await saveButton.click(); expect((await r).status()).toBe(200)`. This is the canonical Playwright pattern for save-flow determinism.
- **Files modified:** apps/web/e2e/wochentage.spec.ts (three save sites: bootstrap, Sa-save, cleanup).
- **Verification:** WOCH-01 green on three consecutive isolated runs.
- **Committed in:** 80e70ec (Task 2 commit).
