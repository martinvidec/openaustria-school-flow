## Deviations from Plan

### Auto-fixed during execution

**1. [Rule 1 — Spec bug] DnD COLLISION test was measuring target bbox before drag-start**

- **Found during:** First end-to-end run of the spec after committing.
- **Symptom:** REGRESSION-DND-COLLISION failed because the lesson moved to (WEDNESDAY, period 5) instead of (THURSDAY, period 5). The Aenderungsverlauf panel + dnd-kit announcement confirmed the actual drop target was Wednesday.
- **Root cause:** `apps/web/src/routes/_authenticated/admin/timetable-edit.tsx:411` hides the Aenderungsverlauf side panel during drag (`!activeLesson` gate). When the panel is removed, the main grid area expands horizontally and CSS Grid auto-sizing redistributes the freed space across day columns. The pre-drag bbox of the THURSDAY/period 5 cell therefore points to a different on-screen position than the cell occupies during drag.
- **Fix:** Defer target-cell `boundingBox()` lookup until AFTER `mouse.down` + the activation-threshold `mouse.move` that activates the drag. Source bbox is still measured pre-drag (the source cell stays put under `opacity: 0.5`). Also switched the target cursor from a 60%-off-center offset to dead-center for maximum subpixel safety against validation-render drift.
- **Commit:** `e2bb830`

**2. [Rule 3 — Blocker] Standard prisma:seed creates ZERO Room rows**

- **Found during:** Initial spec authoring (prior session, included in first fixture commit `bc298f6`).
- **Symptom:** TimetableLesson FK constraint requires a `roomId`, but `findFirst` on Room returned null for the seed school.
- **Fix:** Fixture self-provisions an `e2e-fixture-room-${ts}` Room when none exists for the school, captures its id in `fixtureRoomId`, and `cleanupTimetableRun()` deletes it. Live workflow introduces rooms via the Schuladmin Console UI, not via seed.
- **Why not a real bug:** This is by design — see `apps/api/prisma/seed.ts` (no Room creation) + the Schuladmin UI room creation path. Documenting as a deviation because it diverges from the plan's assumption ("seed has rooms").

**3. [Rule 3 — Blocker] Test DBs may have only MONDAY active in school_days**

- **Found during:** Initial spec authoring (prior session, fixture commit `bc298f6`).
- **Symptom:** The timetable view filters `school.schoolDays where isActive: true` (apps/api/src/modules/timetable/timetable.service.ts:272). If THURSDAY isn't active in the test DB, the grid never renders a THURSDAY column and the spec can't drop a lesson into it.
- **Fix:** Fixture upserts MON–FRI to `isActive: true` and remembers which days it had to flip in `reactivatedDays`. Cleanup flips them back to `false` so the fixture leaves no residue (avoids polluting the test DB with a five-day week if the developer's environment was three-day).

**4. [Rule 3 — Workaround] useClasses() bug forces teacher-perspective drive**

- **Found during:** Initial spec authoring (prior session).
- **Symptom:** `useClasses()` in `apps/web/src/hooks/useTimetable.ts:91` calls `/api/v1/classes` WITHOUT the required `?schoolId=...` query param. The API returns 404 → `classes = []` → no "Klassen" group ever renders in the perspective dropdown. The spec cannot pick a class perspective.
- **Workaround:** Drive via the teacher perspective (which works today). The fixture pins to `kc-lehrer-teacher` (Maria Mueller, stable seed id) and exposes `teacherDisplayName` so the spec can deterministically select "Mueller Maria" in the perspective selector. The three FIXes under test are perspective-agnostic (collision detection, body shape, drag transform are all orthogonal to perspective).
- **Logged as follow-up** — see "Follow-ups" below.

**5. [Rule 1 — Plan literal correction] SolveStatus enum value is `COMPLETED`, not `COMPLETE`**

- **Found during:** Initial fixture authoring (prior session).
- **Symptom:** Plan literal `status: 'COMPLETE'` would have failed at fixture-execution time.
- **Fix:** Use `'COMPLETED'` per `schema.prisma:689`. Documented inline in the fixture.
