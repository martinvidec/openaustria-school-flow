**Fix 1 — apiFetch Content-Type (apps/web/src/lib/api.ts, lines 19-22):**

Change the Content-Type condition from:
```
if (options?.method && options.method !== 'GET' && !headers.has('Content-Type'))
```
to:
```
if (options?.body && !headers.has('Content-Type'))
```

This ensures Content-Type: application/json is only set when a body is actually present. DELETE requests with no body will no longer trigger Fastify's FST_ERR_CTP_EMPTY_JSON_BODY (400) error. This fixes UAT test 19 (resource delete) and the latent bug in room booking cancellation DELETE.

**Fix 2 — useRooms pagination unwrap (apps/web/src/hooks/useTimetable.ts, lines 107-117):**

The room listing API at `/api/v1/schools/:schoolId/rooms` returns a paginated response `{ data: Room[], meta: {...} }`. useTeachers and useClasses both unwrap with `const items = json.data ?? json` and then `.map()` to EntityOption[]. useRooms does neither.

Replace the queryFn body with the same pattern used by useTeachers/useClasses:
```typescript
const res = await apiFetch(`/api/v1/schools/${schoolId}/rooms`);
if (!res.ok) throw new Error('Failed to load rooms');
const json = await res.json();

  id: r.id,
  name: r.name,
}));
```

This ensures PerspectiveSelector receives an EntityOption[] for rooms, and `rooms.length > 0` correctly evaluates, showing the Raeume group in the admin perspective selector dropdown.

**Fix 1 — Exclude lessonId from request body (apps/web/src/hooks/useTimetableEdit.ts, lines 19-23):**

The backend MoveLessonDto has `forbidNonWhitelisted: true` and only accepts `targetDay`, `targetPeriod`, `targetRoomId`. The frontend sends the full `MoveLessonRequest` (which includes `lessonId`) as the body, causing a 422 Unprocessable Entity on every move attempt.

Change the mutationFn to destructure out lessonId:
```typescript
mutationFn: async (dto: MoveLessonRequest): Promise<TimetableViewLesson> => {
  const { lessonId, ...moveBody } = dto;
  const res = await apiFetch(
    `/api/v1/schools/${schoolId}/timetable/lessons/${lessonId}/move`,
    { method: 'PATCH', body: JSON.stringify(moveBody) },
  );
```

This sends only `{ targetDay, targetPeriod, targetRoomId? }` in the body while using `lessonId` only in the URL path.

**Fix 2 — Replace closestCenter with pointerWithin (apps/web/src/routes/_authenticated/admin/timetable-edit.tsx, line 331):**

The `closestCenter` collision detection algorithm calculates distance from the center of the dragged element to the center of droppable elements. Because DraggableLesson applies CSS.Translate to the original element (shifting its bounding box), the "center" of the dragged item is not where the cursor is. This causes the highlighted droppable cell to not match the mouse position.

Replace `closestCenter` with `pointerWithin` which uses the pointer position directly:

1. In the import statement (line 10), replace `closestCenter` with `pointerWithin`:
   ```typescript
   import {
     DndContext,
     ...
     pointerWithin,
     ...
   } from '@dnd-kit/core';
   ```

2. In the DndContext component (line 331), change:
   ```
   collisionDetection={closestCenter}
   ```
   to:
   ```
   collisionDetection={pointerWithin}
   ```

**Fix 3 — Remove CSS.Translate from DraggableLesson (apps/web/src/components/dnd/DraggableLesson.tsx, lines 44-47):**

The CSS.Translate.toString(transform) on the original element moves it during drag, which:
- Confuses @dnd-kit's collision detection geometry (original element shifts out of position)
- Creates visual artifacts (the original moves AND there is a DragOverlay ghost)

Replace the style object to only control opacity, removing the transform entirely:
```typescript
const style: React.CSSProperties = {
  opacity: isDragging ? 0.5 : 1,
};
```

The DragOverlay component (already implemented) provides the visual drag ghost that follows the cursor. The original element stays in place with reduced opacity as a "source position" indicator.
