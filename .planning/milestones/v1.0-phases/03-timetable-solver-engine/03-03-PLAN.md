**Hard constraint (1 new):**

1. `roomTypeRequirement` (D-14): If lesson.requiredRoomType != null, the assigned room must match.
```java
Constraint roomTypeRequirement(ConstraintFactory cf) {
    return cf.forEach(Lesson.class)
        .filter(lesson -> lesson.getRequiredRoomType() != null)
        .filter(lesson -> !lesson.getRequiredRoomType().equals(lesson.getRoom().getRoomType()))
        .penalize(HardSoftScore.ONE_HARD)
        .asConstraint("Room type requirement");
}
```

**Soft constraints (7 new):**

2. `noSameSubjectDoubling` (TIME-02): Penalize two non-consecutive lessons of the same subject on the same day for the same class. Use forEachUniquePair with equal(classId), equal(subjectId), equal(timeslot.dayOfWeek), then filter out pairs that ARE consecutive (which are valid double periods). Helper method `areConsecutive(l1, l2)` checks if timeslot1.nextTimeslotId equals timeslot2.id or vice versa.

3. `balancedWeeklyDistribution` (TIME-02): Use groupBy(classId, subjectId, ConstraintCollectors.loadBalance(lesson -> timeslot.dayOfWeek)) to penalize uneven distribution. If loadBalance is not available in this Timefold version, use countDistinct days and penalize based on unfairness formula: for each class-subject, count lessons per day, compute variance from ideal (totalLessons / schoolDays), penalize deviation.

4. `maxLessonsPerDay` (TIME-02): Group by classId + dayOfWeek, count lessons, penalize each lesson beyond 8 (configurable soft, representing excessive workload per day for students).

5. `preferDoublePeriod` (D-05, TIME-04): For lessons where preferDoublePeriod=true, reward when paired with another lesson of same subject+class on the same day in consecutive periods. If a double-period-preferred lesson has NO consecutive partner, penalize it. Use the ifNotExists pattern from the research:
```java
Constraint preferDoublePeriod(ConstraintFactory cf) {
    return cf.forEach(Lesson.class)
        .filter(Lesson::isPreferDoublePeriod)
        .ifNotExists(Lesson.class,
            Joiners.equal(Lesson::getSubjectId),
            Joiners.equal(Lesson::getClassId),
            Joiners.equal(l -> l.getTimeslot().getDayOfWeek()),
            Joiners.filtering((l1, l2) ->
                (l1.getTimeslot().getNextTimeslotId() != null &&
                 l1.getTimeslot().getNextTimeslotId().equals(l2.getTimeslot().getId())) ||
                (l2.getTimeslot().getNextTimeslotId() != null &&
                 l2.getTimeslot().getNextTimeslotId().equals(l1.getTimeslot().getId()))))
        .penalize(HardSoftScore.ONE_SOFT)
        .asConstraint("Prefer double periods");
}
```

6. `homeRoomPreference` (D-13): If lesson.homeRoomId != null and the assigned room.id does not match, penalize soft.

7. `minimizeRoomChanges` (D-15): For each pair of consecutive lessons for the same class on the same day, if they are in different rooms AND neither requires a special room type (i.e., both could be in a regular Klassenzimmer), penalize the room change.

8. `preferMorningForMainSubjects` (general pedagogical quality): Main subjects (lesson.requiredRoomType == null, representing non-Fachraum subjects like Mathe, Deutsch) penalized for being scheduled after period 6 (afternoon). Lighter penalty.

3. A/B week constraint logic: The A/B week handling is implicit in the domain model. When A/B mode is enabled:
   - NestJS generates timeslots for both weeks (with weekType A and B)
   - Lessons with weekType "BOTH" must appear in both weeks at the same slot
   - Lessons with weekType "A" or "B" only exist in their respective week

   No special constraint is needed IF NestJS correctly duplicates "BOTH" lessons and assigns them to matching week timeslots. However, for safety, add a verification helper that the SolverInputService (Plan 05) will use: a static utility method in Lesson.java:
   ```java
   /**
    * For A/B week mode: returns true if this lesson is compatible with the given timeslot's week type.
    * "BOTH" lessons can go in A or B timeslots.
    * "A" lessons can only go in "A" or "BOTH" timeslots.
    */
   public boolean isWeekCompatible(SolverTimeslot slot) {
       if ("BOTH".equals(this.weekType) || "BOTH".equals(slot.getWeekType())) return true;
       return this.weekType.equals(slot.getWeekType());
   }
   ```
   NOTE: The actual A/B week constraint filtering is handled by the value range provider -- only compatible timeslots are offered to each lesson. This is more efficient than a constraint (avoids search space explosion per Pitfall 1).
