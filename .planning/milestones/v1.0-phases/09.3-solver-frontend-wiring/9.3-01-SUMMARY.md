## Decisions Made

- **createSolverSocket signature is `(schoolId: string)` not `(jwt: string)`** - The backend `TimetableGateway` (namespace `solver`) reads `client.handshake.query.schoolId` and joins clients into `school:{schoolId}` rooms. There is no per-user JWT scoping on this namespace -- progress events are school-scoped broadcasts, not user-scoped. Matching the gateway contract makes the factory trivially compatible with `createTimetableSocket`/`createClassbookSocket`. (Plan draft assumed JWT auth -- corrected during execution; see Deviations below.)
- **useSolverSocket is page-scoped, not layout-scoped** - Following the Phase 5 `useClassbookSocket` decision: solver events are only relevant while the admin is on the solver page. Mounting at `_authenticated` layout would keep the socket alive for every authenticated request which is wasteful for a feature admins hit occasionally.
- **solve:complete payload fields** - `runId, status, hardScore, softScore, elapsedSeconds` match the exact object constructed by `SolverCallbackController.handleCompletion` in `timetable.controller.ts`. The plan's example draft used `lessonsCount` which does not exist in the real payload -- corrected during execution.
- **POST body is empty `{}`** - `StartSolveDto` makes both `maxSolveSeconds` and `constraintWeights` optional; the backend default is 300 seconds (5 min) per D-09. Exposing constraint-weight tuning in the UI is explicitly deferred to v1.1 per 9.3-CONTEXT.md "Deferred Ideas".
- **Generate button disabled while progress is non-null** - Prevents admins from queueing multiple concurrent solves which would contend for the single solver sidecar. Matches the "isRunning" guard pattern from Phase 3's sidecar client.
- **CalendarCog over Sparkles** - Plan allowed either. `CalendarCog` reads as "calendar configuration" which signals the automated-generation feature more literally than `Sparkles`.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Corrected createSolverSocket signature from `(jwt)` to `(schoolId)`**

- **Found during:** Task 1 read-first gate (reading `timetable.gateway.ts` to confirm namespace + event names)
- **Issue:** The plan's example code showed `createSolverSocket(jwt: string)` with `auth: { token: jwt }` and a namespace URL of `${getSocketBaseUrl()}/solver`. But the backend `TimetableGateway` routes clients into `school:{schoolId}` rooms via `client.handshake.query.schoolId` -- there is no per-user JWT scoping on this namespace. Implementing the plan verbatim would have produced a client that connects but never receives progress events because it never joins the school room. Additionally `getSocketBaseUrl()` is not a function that exists in `socket.ts` -- the existing factories use `API_URL` from `import.meta.env.VITE_API_URL`.
- **Fix:** Rewrote the factory as `createSolverSocket(schoolId: string)` passing `{ query: { schoolId }, auth: { token: keycloak.token }, transports: ['websocket', 'polling'] }` -- identical shape to `createTimetableSocket`. Also added `getSolverSocket` and a race-safe `disconnectSolverSocket(socket?)` that matches the `disconnectNotificationSocket` signature so the hook can dispose the exact instance it captured on mount.
- **Files modified:** `apps/web/src/lib/socket.ts`, `apps/web/src/hooks/useSolverSocket.ts`
- **Verification:** `pnpm --filter @schoolflow/web exec tsc --noEmit` exits 0; `grep -c "createSolverSocket" socket.ts` = 1, `grep -c "createSolverSocket" useSolverSocket.ts` = 2.
- **Committed in:** `d30d0e8` (Task 1)

**2. [Rule 1 - Bug] Corrected SolveCompleteEvent payload shape**

- **Found during:** Task 1 read-first gate (reading `timetable.controller.ts` SolverCallbackController)
- **Issue:** The plan's example hook declared `SolveCompleteEvent { runId, hardScore, softScore, lessonsCount }`. The real payload emitted by `SolverCallbackController.handleCompletion` -> `timetableGateway.emitComplete` is `{ runId, status, hardScore, softScore, elapsedSeconds }`. No `lessonsCount` field exists -- per the gateway comment, the full lesson list is intentionally NOT sent via WebSocket (client fetches via REST afterwards).
- **Fix:** Updated `SolveCompleteEvent` in `useSolverSocket.ts` to match the real payload, and updated the toast description and the "Letztes Ergebnis" card in `solver.tsx` to render `status`, `hardScore`, `softScore`, `elapsedSeconds` (plus `runId`).
- **Files modified:** `apps/web/src/hooks/useSolverSocket.ts`, `apps/web/src/routes/_authenticated/admin/solver.tsx`
- **Verification:** TSC exits 0; all rendered fields type-check against the corrected interface.
- **Committed in:** `d30d0e8` / `be59079`

**3. [Rule 1 - Bug] Corrected SolveProgressEvent payload shape**

- **Found during:** Task 1 read-first gate (reading `dto/solve-progress.dto.ts`)
- **Issue:** The plan's example declared `SolveProgressEvent { runId, percent, hardScore, softScore }`. The real `SolveProgressDto` has NO `percent` field -- it has `runId, hardScore, softScore, elapsedSeconds, remainingViolations, improvementRate, scoreHistory`. Timefold solvers do not expose a linear "percent done" signal because the solve time is bounded by `maxSolveSeconds` and the improvement is monotonic-ish but non-linear.
- **Fix:** Extended `SolveProgressEvent` to mirror `SolveProgressDto` exactly, plus added `SolveViolationGroup` and `SolveScoreHistoryEntry` helper types mirroring `ViolationGroupDto` / `ScoreHistoryEntryDto`. The solver page renders elapsed seconds (instead of a fake percent bar), the improvement rate, and the top-5 remaining violation groups which collectively tell the admin "what's the solver working on and how close is it to feasible".
- **Files modified:** `apps/web/src/hooks/useSolverSocket.ts`, `apps/web/src/routes/_authenticated/admin/solver.tsx`
- **Verification:** TSC exits 0; progress card renders all fields from the type.
- **Committed in:** `d30d0e8` / `be59079`

**4. [Rule 3 - Blocking] Corrected useAuth import path in the plan's example**

- **Found during:** Task 2 drafting of `solver.tsx`
- **Issue:** The plan's example imported `useAuth` from `@/lib/auth`. This file does not exist in the codebase -- `useAuth` is a custom hook at `@/hooks/useAuth`, and its return shape is `{ user: { id, roles, ... }, isAuthenticated, login, logout }` (no `jwt` field -- the JWT is pulled from `keycloak.token` via `apiFetch`).
- **Fix:** Because Task 1's fix made `useSolverSocket` take `schoolId` instead of `jwt`, the `useAuth` call becomes unnecessary in `solver.tsx` -- the page uses `useSchoolContext((s) => s.schoolId)` directly. The `apiFetch` wrapper handles Keycloak token refresh + Authorization header automatically. Removing the `useAuth` dependency simplified the page and avoided the broken import entirely.
- **Files modified:** `apps/web/src/routes/_authenticated/admin/solver.tsx`
- **Verification:** TSC exits 0; page renders without any `useAuth` dependency.
- **Committed in:** `be59079`
