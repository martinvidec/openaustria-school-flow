    **Step 1: Create usePushSubscription hook**
    At `apps/web/src/hooks/usePushSubscription.ts`:

    State tracking:
    - `permissionState`: 'default' | 'granted' | 'denied' -- from `Notification.permission`
    - `isSubscribed`: boolean -- whether a push subscription exists on the current SW registration
    - `isLoading`: boolean -- during subscribe/unsubscribe operations
    - `error`: string | null -- error message from failed operations

    Functions:
    - `subscribe()`:
      1. Check `'Notification' in window && 'serviceWorker' in navigator` -- if not, set error and return (Pitfall 1: iOS Safari pre-install)
      2. Request notification permission: `await Notification.requestPermission()`
      3. If denied, update permissionState and return
      4. Fetch VAPID public key: `GET /api/v1/push/vapid-key` (no auth needed)
      5. Get SW registration: `await navigator.serviceWorker.ready`
      6. Subscribe to push: `await registration.pushManager.subscribe({ userVisibleOnly: true, applicationServerKey: urlBase64ToUint8Array(publicKey) })`
      7. Send subscription to backend: `POST /api/v1/push-subscriptions` with `{ endpoint, keys: { p256dh, auth } }` extracted from PushSubscription.toJSON()
      8. Update isSubscribed state

    - `unsubscribe()`:
      1. Get current push subscription: `await registration.pushManager.getSubscription()`
      2. If exists, call `subscription.unsubscribe()` on the browser side
      3. Delete from backend: `DELETE /api/v1/push-subscriptions` with `{ endpoint }`
      4. Update isSubscribed state

    Helper `urlBase64ToUint8Array(base64String)`: Standard base64url to Uint8Array conversion for VAPID applicationServerKey.

    On mount: Check `Notification.permission` and current push subscription state. Update permissionState and isSubscribed accordingly.

    Return: `{ permissionState, isSubscribed, isLoading, error, subscribe, unsubscribe }`

    **Step 2: Create PushNotificationSettings component**
    At `apps/web/src/components/settings/PushNotificationSettings.tsx`:

    Per 09-UI-SPEC Push Notification Settings Card:
    - Card wrapper with heading "Push-Benachrichtigungen" (20px semibold) and description text from copywriting contract
    - Description: "Erhalten Sie Benachrichtigungen fuer Stundenplanaenderungen, neue Nachrichten und Abwesenheits-Alerts direkt auf Ihrem Geraet."

    Three visual states based on `usePushSubscription()`:

    **State A: Not subscribed (permissionState === 'default' or 'granted' but !isSubscribed)**
    - Primary Button: "Benachrichtigungen aktivieren"
    - Status text (12px, muted): "Benachrichtigungen deaktiviert"
    - On click: calls `subscribe()`

    **State B: Subscribed (isSubscribed === true)**
    - Outline Button with success indicator: "Benachrichtigungen aktiv"
    - Status text (12px, muted, success color): "Benachrichtigungen aktiv" with CheckCircle icon
    - On click: calls `unsubscribe()`

    **State C: Permission denied (permissionState === 'denied')**
    - Warning card with `--color-warning` left border (4px)
    - Heading: "Benachrichtigungen blockiert"
    - Body: "Benachrichtigungen wurden im Browser blockiert. Bitte aktivieren Sie diese in den Browser-Einstellungen und laden die Seite neu."
    - Button disabled

    **Error state:** Inline error text (12px, destructive color) below button: "Benachrichtigungen konnten nicht aktiviert werden. Bitte versuchen Sie es erneut." with "Erneut versuchen" link.

    **Loading state:** Button shows spinner and is disabled during subscribe/unsubscribe.
