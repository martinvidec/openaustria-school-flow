# Phase 9 — UI Design Contract

> Visual and interaction contract for Phase 9: Mobile, PWA & Production Readiness. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Phase classification: SIXTH FRONTEND PHASE (FINAL).** All design tokens (spacing, typography, base colors, semantic colors, subject palette, change indicator colors) are inherited verbatim from Phase 4/5/6/7/8 UI-SPECs. Phase 9 introduces NO new design tokens. This phase delivers: (1) responsive audit and fixes across all existing pages, (2) PWA shell with service worker, offline timetable, and install prompt, (3) Web Push notification opt-in UI in settings, (4) service worker update toast notification, and (5) offline indicator banner. Production infrastructure work (backup scripts, health endpoints, Docker production profile) is backend-only with no frontend UI.

## Spacing Scale

Declared values inherited from Phase 4 UI-SPEC (multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding, offline banner icon-to-text gap, PWA install button icon gap |
| sm | 8px | Compact element spacing, push notification toggle internal padding, update toast icon spacing |
| md | 16px | Default element spacing, settings section card padding, offline banner internal padding |
| lg | 24px | Section padding, push notification settings card padding, PWA install banner padding |

| 2xl | 48px | Major section breaks, page-level vertical padding |
| 3xl | 64px | Page-level spacing, content max-width gutters |

Exceptions (Phase 9 specific):
- Touch targets on mobile: minimum 44x44px (WCAG 2.5.5 AAA). Applies to ALL interactive elements audited during responsive pass -- timetable cells, attendance quick-tap buttons, message list items, navigation items, form controls, dialog buttons.
- Offline banner height: 40px fixed. Sticks to top of viewport below AppHeader (56px header height). Banner must not overlap content -- pushes main content down.
- PWA install banner: 56px height on mobile, 48px on desktop. Positioned at bottom of viewport as a fixed bar with 16px horizontal padding.
- Service worker update toast: uses Sonner toast component (already integrated in `__root.tsx`). No custom sizing needed.

## Responsive Audit Contract

Phase 9 MOBILE-01 requires all existing pages to be fully usable on smartphone and tablet. The following is the prescriptive audit scope with specific layout rules per breakpoint.

### Breakpoints (Tailwind 4 defaults)

| Breakpoint | Min Width | Usage |
|------------|-----------|-------|
| (base) | 0px | Phone portrait (320-639px) |
| sm | 640px | Phone landscape / small tablet |
| md | 768px | Tablet portrait |
| lg | 1024px | Tablet landscape / desktop |
| xl | 1280px | Desktop wide |

### Page-by-Page Responsive Rules

**1. Timetable (`/timetable`)**
- base/sm: Force day view. Hide week toggle. Swipe between days via day selector tabs (horizontal scroll). Period rows at 48px height (reduced from 56px). Subject abbreviation only (no teacher/room in cell, show on tap).
- md: Day or week view available. Week view shows 5 columns at compressed width. Full cell content (subject + teacher + room).
- lg+: Full week view with generous column widths. All cell content visible.

**2. Classbook Lesson Detail (`/classbook/$lessonId`)**
- base/sm: Tabs stack as full-width horizontal scroll tabs. AttendanceGrid renders as vertical list (one student per row, tap to cycle status) instead of grid. GradeMatrix scrolls horizontally with frozen student name column.
- md: AttendanceGrid as 2-column layout. GradeMatrix with horizontal scroll.
- lg+: Full grid layout. GradeMatrix fits without scroll for typical class sizes.

**3. Messaging (`/messages`)**
- base/sm: List-only view. Tapping a conversation navigates to `/messages/$conversationId` (separate route already exists). Back button returns to list. ComposeDialog is full-screen on mobile.
- md+: Side-by-side list-detail split view with `?id=` search param (already implemented per Phase 7 decision).

**4. Admin Substitutions (`/admin/substitutions`)**
- base/sm: Tabs stack vertically. AbsenceForm fields stack single-column. CandidateList cards stack vertically. FairnessStatsPanel table scrolls horizontally.
- md+: Current layout preserved.

**5. Room Availability (`/rooms`)**
- base/sm: Room availability grid scrolls horizontally with frozen time column. Room booking dialog is full-screen modal.
- md+: Current layout preserved.

**6. Import Wizard (`/admin/import`)**
- base/sm: Wizard steps show as horizontal progress dots (not step labels). ImportColumnMapper renders as stacked cards (one per column) instead of table. ImportDryRunPreview table scrolls horizontally.
- md+: Current layout preserved with step labels and table column mapper.

**7. Excuses (`/excuses`)**
- base/sm: ExcuseForm fields stack single-column. FileUploadField spans full width.
- md+: Current layout preserved.

**8. Statistics Absence (`/statistics/absence`)**
- base/sm: AbsenceStatisticsPanel table scrolls horizontally. Date range pickers stack vertically.
- md+: Current layout preserved.

**9. Admin Resources (`/admin/resources`)**
- base/sm: ResourceList table scrolls horizontally. Edit dialog is full-screen.
- md+: Current layout preserved.

**10. Settings (`/settings`)**
- base/sm: Settings sections (iCal, Push, PWA) stack vertically as full-width cards. All form controls span full width.
- md+: Max-width 640px centered container for settings cards.

### Global Responsive Rules

| Rule | Specification |
|------|--------------|
| Main content padding | `p-4` at base, `p-6` at md+ (already in `__root.tsx`) |
| Dialog sizing | Full-screen (`w-full h-full`) at base/sm, centered max-width at md+ |
| Sidebar | Hidden at base (hamburger menu in AppHeader), visible at sm+ (already implemented via MobileSidebar) |
| Touch targets | Minimum 44x44px for all interactive elements at base/sm breakpoints |
| Tables | Horizontal scroll wrapper (`overflow-x-auto`) at base/sm. No horizontal scroll at md+ for tables with <= 5 columns |
| Form layouts | Single column at base/sm. Two columns where appropriate at md+ |
| Toasts | Bottom-center on mobile (avoid top-right overlap with notch). Top-right on desktop (current Sonner config) |

## PWA UI Components

### Offline Banner

| Property | Value |
|----------|-------|
| Position | Fixed, below AppHeader (top: 56px), full width, z-index 40 |
| Height | 40px |
| Background | `hsl(38 92% 50% / 0.10)` |
| Border | 1px solid `hsl(38 92% 50% / 0.30)` on bottom edge |

### PWA Install Banner

| Property | Value |
|----------|-------|

| Shadow | `shadow-lg` (elevated above content) |
| Content | SchoolFlow icon (24px) + description text (14px body) + "App installieren" button (primary) + "Nicht jetzt" button (ghost) |

## Interaction States

### Offline/Online Transitions

| State | Visual |
|-------|--------|
| Online | No banner. Normal operation. |
| Transitioning offline | OfflineBanner slides down from below AppHeader (150ms ease-out). Main content shifts down 40px. |
| Offline | OfflineBanner visible. Timetable page serves cached today view. Other pages show inline "Offline" messages where data cannot load. |
| Transitioning online | OfflineBanner slides up and hides (150ms ease-in). Main content shifts up. TanStack Query triggers background refetch of stale queries. |

### Push Notification Permission Flow

| Step | User Action | System Response |
|------|------------|-----------------|
| 1 | Opens settings page | Push card shows current state (enabled/disabled/blocked) |
| 2 | Clicks "Benachrichtigungen aktivieren" | Browser permission prompt appears |
| 3a | Grants permission | Button changes to "Benachrichtigungen aktiv" (outline variant). Subscription sent to backend via POST `/api/v1/push-subscriptions`. Success toast: "Benachrichtigungen aktiviert." |
| 3b | Denies permission | Card transitions to "blocked" state with warning styling and instructions |
| 3c | Dismisses prompt | Button returns to "Benachrichtigungen aktivieren" (no change) |
| 4 | Clicks active toggle to disable | Subscription removed from backend via DELETE. Button returns to "Benachrichtigungen aktivieren". |

### PWA Install Flow

| Step | User Action | System Response |
|------|------------|-----------------|
| 1 | Visits app, `beforeinstallprompt` fires | PwaInstallBanner appears at bottom of viewport |
| 2a | Clicks "App installieren" | Browser install prompt appears. If accepted, banner hides, settings card shows installed state. |
| 2b | Clicks "Nicht jetzt" | Banner hides for this session. Settings card still shows install button. |
| 3 | Opens settings when already installed | PWA install card shows "SchoolFlow ist bereits als App installiert." |
