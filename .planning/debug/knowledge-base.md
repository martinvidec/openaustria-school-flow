## dnd-constraint-validation — Drag-and-drop on timetable edit page snapped back, wrong cell highlighted, no yellow/red feedback
- **Date:** 2026-04-25
- **Error patterns:** drag, drop, dnd, snap-back, snap back, lessonId, MoveLessonDto, forbidNonWhitelisted, 422, ValidationPipe, closestCenter, pointerWithin, collisionDetection, DraggableLesson, CSS.Translate, transform, DragOverlay, validate-move, hardViolations, softWarnings, timetable-edit, useMoveLesson, useDragConstraints
- **Root cause:** Three concurrent root causes: (1) `useMoveLesson` sent `lessonId` in PATCH body, but `MoveLessonDto` whitelist excludes it and `ValidationPipe` has `forbidNonWhitelisted: true` → automatic 422 on every move; (2) `DndContext` used `closestCenter` collision detection, which selected the geometrically-nearest empty droppable instead of the cell under the pointer; (3) `DraggableLesson` applied `CSS.Translate.toString(transform)` to the original element while a `DragOverlay` was also active, shifting its bbox and worsening collision targeting.
- **Fix:** (1) Destructure `const { lessonId, ...moveBody } = dto;` and send `JSON.stringify(moveBody)` so `lessonId` stays in the URL only; (2) Switch `collisionDetection` from `closestCenter` to `pointerWithin`; (3) Remove the CSS transform from the original `DraggableLesson` and only set `opacity` (let `DragOverlay` handle the visual preview). All three fixes shipped together in commit `de9ee2b` "fix(04-13): fix drag-and-drop lesson moves".
- **Files changed:** apps/web/src/hooks/useTimetableEdit.ts, apps/web/src/routes/_authenticated/admin/timetable-edit.tsx, apps/web/src/components/dnd/DraggableLesson.tsx

## useclasses-missing-schoolid — Klassen perspective group missing in PerspectiveSelector with no error toast (silent-omission pattern)
- **Date:** 2026-04-25
- **Error patterns:** useClasses, classes, Klassen, perspective, PerspectiveSelector, SelectGroup, SelectLabel, /api/v1/classes, schoolId, 404, NotFoundException, ClassService.findAll, ClassListQueryDto, silent omission, missing group, dropdown missing, useTimetable, EntityOption, paginated unwrap, data default empty, length > 0 conditional render, timetable-edit, admin perspective, no error toast
- **Root cause:** `useClasses()` in `apps/web/src/hooks/useTimetable.ts` fetched `/api/v1/classes` with NO query string. The class API requires `?schoolId=...` (`ClassService.findAll` throws `NotFoundException` → HTTP 404 when the param is missing). The TanStack Query error was swallowed by the consumer's `const { data: classes = [] } = useClasses(...)` default, and `PerspectiveSelector` renders the Klassen `SelectGroup` only behind `{classes.length > 0 && <SelectGroup>...}` — so the entire group disappeared with zero visible failure. This is a **silent-omission pattern**: hook misses required query param + consumer destructures `data ?? []` + downstream renders behind a `array.length > 0 && <render>` guard. Three layers each independently silent → bug invisible to users AND developers (only signal is the 404 in the network panel).
- **Fix:** Build `URLSearchParams` with `schoolId + page=1 + limit=500` and request `/api/v1/classes?<params>`. Response unwrap (`json.data ?? json`) and `{id, name}` mapping were already correct — only the request URL was broken. Single-line fix shipped in commit `d76b5a3` with regression coverage in `apps/web/src/hooks/__tests__/useTimetable.spec.ts` (3 unit tests) and `apps/web/e2e/admin-timetable-edit-perspective.spec.ts` (E2E asserting the Klassen SelectLabel renders + a seeded class is selectable + selecting one mounts the grid).
- **Files changed:** apps/web/src/hooks/useTimetable.ts, apps/web/src/hooks/__tests__/useTimetable.spec.ts (new), apps/web/e2e/admin-timetable-edit-perspective.spec.ts (new), apps/web/e2e/fixtures/timetable-run.ts (comment-only update)
- **Related finding (separate debug session worth opening):** `useTeachers()` in the same file has the IDENTICAL structural bug (no `schoolId` on `/api/v1/teachers`), but does NOT 404 because `TeacherService.findAll` does not validate schoolId — Prisma treats `where: { schoolId: undefined }` as "no filter" and returns teachers from ALL tenants. Masked tenant-isolation hole. Out of scope for this session per strict-scope directive; should be a separate `/gsd:debug` task that also audits `TeacherService.findAll` for tenant enforcement. **RESOLVED 2026-04-26 — see `useteachers-tenant-isolation-leak` entry below.**

## useteachers-tenant-isolation-leak — useTeachers + TeacherService.findAll silently returned teachers from ALL schools when schoolId was missing (silent-permissiveness pattern, cross-tenant data leak)
- **Date:** 2026-04-26
- **Error patterns:** useTeachers, teachers, Lehrer, /api/v1/teachers, TeacherService, TeacherService.findAll, teacher.service, teacher.controller, SchoolPaginationQueryDto, IsOptional, schoolId, query.schoolId!, non-null assertion, where schoolId undefined, Prisma undefined strip, silent permissiveness, cross-tenant, tenant isolation, tenant leak, multi-tenant, DSGVO, perspective, PerspectiveSelector, useTimetable, defense in depth, NotFoundException, findAll guard, subject.service, audit findAll
- **Root cause:** Two-layer silent leak. (1) `apps/web/src/hooks/useTimetable.ts` `useTeachers()` at L72 fetched `/api/v1/teachers` with NO query string. (2) `apps/api/src/modules/teacher/teacher.service.ts` `findAll()` had `const where: any = { schoolId };` with NO guard. The controller at `teacher.controller.ts:48` forwarded `query.schoolId!` from a `SchoolPaginationQueryDto` where `schoolId` is `@IsOptional()` — the non-null assertion was a TS-level lie. When `schoolId` was undefined, Prisma stripped the key from the where clause (`findMany({ where: { schoolId: undefined } })` is treated as "no filter") and returned teachers from EVERY school in the database. Any authenticated user with `subject: teacher / read` (admin OR teacher) could enumerate teachers across all tenants by hitting `/admin/timetable-edit` — DSGVO-relevant cross-tenant breach masked as a normal-looking populated UI (no toast, no error, no log signal).
- **Fix:** Defense in depth across both layers. **Backend (`d7e1c9d`)**: added `if (!schoolId) throw new NotFoundException('schoolId query parameter is required');` at the top of `TeacherService.findAll`, mirroring the canonical `ClassService.findAll` guard. Spec lock-down includes `expect(mockPrisma.teacher.findMany).not.toHaveBeenCalled()` — the gold-standard leak-prevention contract that proves the guard short-circuits before Prisma. **Frontend (`3e9de88`)**: `useTeachers` now builds `URLSearchParams` with `schoolId + page=1 + limit=500` and requests `/api/v1/teachers?<params>`. 3 new unit tests mirror the `useClasses` pattern (URL contains `?schoolId=`, paginated `{ data, meta }` envelope maps to `EntityOption[]`, no fetch when schoolId undefined). The backend guard is the authoritative leak closure — even a one-line frontend fix needs a corresponding backend guard because future callers (other hooks, mobile app, plugins, scripts) cannot be trusted to pass the param.
- **Files changed:** apps/api/src/modules/teacher/teacher.service.ts, apps/api/src/modules/teacher/teacher.service.spec.ts, apps/web/src/hooks/useTimetable.ts, apps/web/src/hooks/__tests__/useTimetable.spec.ts
- **Pattern (silent-permissiveness, dual of useclasses-missing-schoolid):** The `useClasses` entry above documents the **silent-omission** variant (`{ data: x = [] }` + `{x.length > 0 && <render>}` → UI element vanishes). This entry is the dual: **silent-permissiveness** — `where: { schoolId: undefined }` → Prisma drops the filter → cross-tenant data returned with HTTP 200. Same root cause (hook misses required query param), opposite symptoms (UI gone vs. UI overpopulated), wildly different severity (UX bug vs. compliance breach). When auditing tenant-scoped endpoints: every `useX(activeSchoolId)`-style hook needs both a request-URL test AND a corresponding backend `if (!schoolId) throw` guard.
- **Audit taxonomy for `findAll(` endpoints (re-usable for future audits):**
  - **A. SAFE (URL-path `:schoolId`)** — Nest `@Param('schoolId')` requires the segment, route 404s on missing path: `resource.controller`, `room.controller`, `school-year.controller`, `constraint-template.controller`, `homework.controller`, `communication/conversation.controller`.
  - **B. SAFE (query DTO + explicit guard)** — Service throws `NotFoundException`/`BadRequestException` when `!schoolId`: `class.service.ts L49-52` (canonical), `parent.service.ts L37-41`, `student.service.ts L65-82`, **and now `teacher.service.ts` (this fix)**.
  - **C. LEAK (query DTO + `@IsOptional` + no service guard)** — The bug pattern. After this fix, exactly ONE remaining instance: **`apps/api/src/modules/subject/subject.service.ts:42` + `subject.controller.ts:45-49`**. Same fix mechanism (one-line guard mirroring TeacherService) — **CLOSED 2026-04-26 in commit `6cf3c94` via `subject-tenant-isolation-leak` debug session.** Category C is now empty across the API (zero remaining instances).
  - **D. INTENTIONALLY OPEN** — Superadmin/global endpoints with no school filter expected: `school.service` (lists all schools), `audit.service`, `exam.controller` (classId-scoped, not schoolId).
- **Related (RESOLVED 2026-04-26):** `apps/api/src/modules/subject/subject.service.ts:42` — identical silent-permissiveness pattern, only other `findAll` instance found in full audit. Closed in commit `6cf3c94` via `subject-tenant-isolation-leak` debug session. Category C is now empty.

## subject-tenant-isolation-leak — SubjectService.findAll silently returned subjects from ALL schools when schoolId was missing (silent-permissiveness pattern, third entry in family, closes Category C audit)
- **Date:** 2026-04-26
- **Error patterns:** useSubjects, subjects, Fächer, Stundentafel, /api/v1/subjects, SubjectService, SubjectService.findAll, subject.service, subject.controller, SchoolPaginationQueryDto, IsOptional, schoolId, query.schoolId!, non-null assertion, where schoolId undefined, Prisma undefined strip, silent permissiveness, cross-tenant, tenant isolation, tenant leak, multi-tenant, DSGVO, Category C, NotFoundException, findAll guard, audit closed
- **Root cause:** Single-layer silent leak. `SubjectService.findAll` at `subject.service.ts:42` had `where: { schoolId }` with no guard; `SubjectController.findAll` at `subject.controller.ts:46` forwarded `query.schoolId!` from a `SchoolPaginationQueryDto` where `schoolId` is `@IsOptional()`. When `schoolId` was undefined, Prisma stripped the key from the where clause and `findMany` returned subjects from EVERY school in the database. Identical mechanic to the just-fixed `useteachers-tenant-isolation-leak` — see that entry for the pattern teaching, taxonomy, and Prisma-strip mechanic.
- **Fix:** Added `if (!schoolId) throw new NotFoundException('schoolId query parameter is required');` at the top of `SubjectService.findAll`, mirroring `TeacherService.findAll` L86-88 verbatim. Three regression spec cases including `expect(mockPrismaService.subject.findMany).not.toHaveBeenCalled()`. **Frontend NOT touched** — `useSubjects` (`apps/web/src/hooks/useSubjects.ts`) was already correct (URLSearchParams + `enabled: !!schoolId`), as was `StundentafelTab`'s `useSubjectsList`. Per playbook directive: no churn for the sake of symmetry. Falsification-tested by temporarily disabling the guard — both new tests fail, restored, all green.
- **Files changed:** apps/api/src/modules/subject/subject.service.ts, apps/api/src/modules/subject/subject.service.spec.ts
- **Pattern closure:** This session closes the Category C audit bucket from the `useteachers-tenant-isolation-leak` taxonomy. After this fix: zero remaining `query.schoolId!` + `@IsOptional` + no-guard sites across `apps/api/src/modules`. New `findAll` PRs must fall in Category A (URL-path `:schoolId`) or B (query DTO + explicit guard) — never C. Reviewers can cite this trio (`useclasses-missing-schoolid` + `useteachers-tenant-isolation-leak` + this entry) when blocking a non-compliant new endpoint.
